# Feature Specification: Playground 页（结构化流式 Agent 演练场）

**Feature Branch**: `001-playground`
**Created**: 2026-04-23
**Status**: Draft
**Input**: User description: "Playground 页 - Spec v1.0 — 输入 prompt → 选择模型 → 看 AI 流式返回结构化 UI 卡片（思考过程 / 工具调用 / 最终答案），而非单一文本块"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 结构化流式对话（Priority: P1）

一位同时使用多个模型的开发者打开 Playground，从顶部下拉里选一个模型，在中间输入框里写一段 prompt，按 cmd+enter 提交。下半部分立刻开始流式渲染：先出现一张"思考过程"时间线卡，接着出现一张"工具调用"表格卡，最后出现一张 markdown 的"最终答案"卡。三张卡片不是等全部到齐才显示，而是**字段一到就亮一块，没到的字段先显示骨架屏**。

**Why this priority**：这是 Playground 的唯一存在理由——展示"结构化输出 Agent"相较"聊天窗口"的价值。没有这段体验，整个页面失去意义。

**Independent Test**：只要实现这一条就能交付 MVP——发送 prompt、看到三种卡片逐字段流式呈现、最后得到完整结构化结果，即视为核心价值达成。

**Acceptance Scenarios**:

1. **Given** 用户已选好模型、输入框里有一段 prompt，**When** 按下 cmd+enter，**Then** 响应区在 2 秒内出现第一张卡片的首个字段（不等整体结果到齐）
2. **Given** 模型正在流式返回中，**When** 某张卡片只收到部分字段，**Then** 已到字段正常渲染，未到字段显示骨架屏，不阻塞其他卡片
3. **Given** 模型返回完成，**When** 用户查看响应区，**Then** 依次看到 "思考过程" timeline 卡、"工具调用" 表格卡、"最终答案" markdown 卡，三者之间留出清晰的视觉分隔

---

### User Story 2 - 多模型横向切换（Priority: P2）

开发者在同一个 prompt 上，先后切换不同厂商的模型（GPT / Claude / Gemini / DeepSeek / Qwen），对比各自在"思考过程"和"最终答案"上的差异。

**Why this priority**：多模型对比是 Playground 相较单模型聊天窗口的差异化卖点，但它依附于 Story 1 —— 没有结构化流式，对比也没意义。

**Independent Test**：在已有 Story 1 的前提下，切换模型后再次发送同一 prompt，能看到不同模型各自产出的结构化卡片，且切换动作本身在 300ms 内完成。

**Acceptance Scenarios**:

1. **Given** 顶部模型选择器展开，**When** 用户查看可选项，**Then** 至少出现 5 个模型选项（覆盖 GPT 5.3 / Claude 4.7 / Gemini 3.1 Flash / DeepSeek V3.2 / Qwen 3.5 Flash 五家主流厂商）
2. **Given** 用户已发送过一次 prompt 并看到结果，**When** 切换模型并重新发送同一 prompt，**Then** 响应区清空后重新流式渲染新模型的结构化结果

---

### User Story 3 - 查看原始结构化数据（Priority: P3）

对结构化输出感兴趣的使用者（教学场景中的学员、做二次调试的开发者），希望看到三张卡片背后真正的 JSON 载荷长什么样。

**Why this priority**：这是"教学属性"加分项，不是核心体验。没有它 Playground 依然成立，但有了它能让使用者理解"卡片 = schema 实例化结果"这一本质。

**Independent Test**：在 Story 1 已可用的前提下，点击"复制原始 JSON"能把完整结构化对象写入剪贴板，粘贴到外部编辑器可见合法 JSON。

**Acceptance Scenarios**:

1. **Given** 流式已完成、三张卡片都已渲染，**When** 用户点击"复制原始 JSON"按钮，**Then** 剪贴板中是合法、可解析的 JSON 字符串，字段与卡片展示一一对应
2. **Given** 流式尚未完成，**When** 用户点击"复制原始 JSON"按钮，**Then** 复制的是当前已到达的部分 JSON（而不是报错，也不是空值）

---

### Edge Cases

- **流式中途网络断开**：用户已看到前两张卡片的部分字段，第三张未开始。系统必须显式给出错误提示（文案 + 图标），并保留已渲染的卡片内容，不清空。
- **模型返回的 JSON 不符合 schema**：系统必须显式给出"结构化输出解析失败"错误，并允许用户一键查看原始返回文本，不静默吞错。
- **用户在流式中途再次点击"发送"或切换模型**：当前流必须先中止，响应区清空，开始新一轮流式。
- **prompt 为空或纯空白**：发送按钮禁用，cmd+enter 无响应。
- **用户在 768px 宽度下访问**：布局可用（不破版、不横向滚动），但在 1280px+ 下才是"推荐体验"。
- **复制 JSON 时剪贴板 API 被浏览器拒绝**（权限、非 https）：给出可见失败反馈，而不是让按钮看起来像成功。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**：页面顶部必须提供一个模型选择器，至少包含 5 个选项，覆盖 GPT 5.3 / Claude 4.7 / Gemini 3.1 Flash / DeepSeek V3.2 / Qwen 3.5 Flash
- **FR-002**：页面中部必须提供一个多行输入框，支持 cmd+enter（或 ctrl+enter）提交，并提供"清空"与"发送"两个按钮
- **FR-003**：当 prompt 为空或仅含空白字符时，"发送"按钮必须处于禁用态，且快捷键不触发提交
- **FR-004**：AI 响应必须以流式方式呈现为一组**结构化卡片**，当前版本定义三类：思考过程（timeline 形态）、工具调用（表格形态）、最终答案（markdown 形态）
- **FR-005**：每张卡片必须支持**字段级渐进渲染**——任一字段一到就立即展示，未到字段显示骨架屏；禁止"整卡等齐再显示"
- **FR-006**：卡片之间必须保持清晰视觉分隔（遵循视觉宪法的 24px 卡片间距）
- **FR-007**：系统必须提供"复制原始 JSON"操作，能把当前响应的结构化数据写入剪贴板；流式中途点击时复制的是已到达的部分快照
- **FR-008**：当后端流式过程发生错误（网络中断、解析失败、超时等），UI 必须显式展示错误文案与图标，禁止静默失败；已渲染的卡片内容保留不清空
- **FR-009**：用户在流式过程中切换模型或再次点击"发送"时，当前流必须先中止，响应区清空后再开始新一轮
- **FR-010**：结构化内容的字段定义必须前后端共用同一份 schema，作为响应区渲染和后端产出的 Single Source of Truth
- **FR-011**：页面在 ≥ 768px 宽度下必须保持可用（不破版、无横向滚动）；在 ≥ 1280px 下提供推荐体验
- **FR-012**：页面所有颜色与间距必须走项目设计 token（@theme 变量与 4px 阶梯），禁止字面量色值或任意间距
- **FR-013**：流式进行中必须有明确的 loading 视觉反馈（骨架屏或渐变闪烁），让用户不会怀疑"是不是卡住了"

### Key Entities *(include if feature involves data)*

- **Model Option**：模型选择器中的一项，承载"显示名 + 厂商 + 可路由标识"，不暴露 API Key 相关细节
- **Prompt Submission**：一次提交动作，包含所选模型与 prompt 文本；同一时刻只允许一条处于"进行中"
- **Structured Response**：一次 AI 响应的结构化结果，由三部分组成——思考过程（按步骤时间线）、工具调用（每行含工具名/入参/出参/耗时）、最终答案（markdown 字符串）；每个字段都允许处于"尚未到达"状态以支撑渐进渲染
- **Stream Error**：流式过程中产生的可见错误，包含错误类别（网络 / 解析 / 超时 / 其他）与展示文案

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**：95% 的提交在 2 秒内出现响应区第一个可见字段（首字段到达时间）
- **SC-002**：字段级渐进渲染覆盖率 100%——任一已到达字段都不被"等齐"逻辑延迟展示
- **SC-003**：流式异常场景下 100% 给出可见错误反馈，零静默失败（通过异常注入回归测试覆盖）
- **SC-004**：在 1280px+ 分辨率下，用户首次使用完成一次"输入 → 选模型 → 看结构化结果"闭环的平均时间 < 30 秒
- **SC-005**：在 768px 分辨率下，核心交互（输入、选模型、发送、查看三张卡片）可达性 100%，无破版
- **SC-006**：三张卡片之间的视觉间距在全部断点下稳定为设计 token 规定的 24px，偏差 0 像素

## Assumptions

- Playground 是**无状态演练场**：本 Spec 不涉及历史对话、收藏、分享；会话一旦离开即销毁（历史由下一个 Spec `RunHistory` 负责）
- API Key 管理不在本 Spec 范围内，沿用项目既有的"浏览器端 Key 存 localStorage + 服务端 Edge 路由代理"模式
- 模型路由层至少需要接入 2 家厂商的网关，其余厂商在模型选择器中可为"已列出但未启用"状态，具体启用哪些由实现阶段决定
- 不渲染 markdown 数学公式（KaTeX 体积成本不划算）
- 不提供对话分享链接
- 首个 token 出现时间 < 2s 的目标依赖网关延迟，系统可控部分必须为"收到首个 token 后立即渲染"，不做额外缓冲
- 桌面浏览器优先；移动端不在本 Spec 范围内（项目级约束）
