**文档编号**：04_interaction_design/t3_hitl_approval

**输入文档**：`03_problem_modeling/problem_model.md`
**阶段覆盖**：高风险条款强制中断 → 人工逐条审批 → 流程恢复 → 报告生成

## 一、阶段概述

### 1.1 触发前提

本文档描述的交互链路在以下前提下被激活：

- Teammate 2 负责的 AI 扫描与分级路由阶段已完成
- `ReviewSession.state` 已转为 `hitl_pending`
- 至少存在一条 `risk_level = high` 且 `human_decision = pending` 的 `ReviewItem`
- LangGraph 工作流已在高风险节点触发 `interrupt`，等待人工恢复信号

### 1.2 阶段目标

| 目标 | 具体描述 |
|------|----------|
| 人工决策不可绕过 | 每条高风险条款必须由审核人员明确执行 Approve / Edit / Reject 操作之一 |
| 防止 Automation Bias | 界面设计强制审核人员阅读原文并填写判断理由，不提供一键批量通过 |
| 操作全程可追溯 | 每个决策操作通过 Checkpointer 持久化，记录操作人、时间、理由 |
| 支持异步跨天恢复 | 中断后可从断点继续，不丢失已完成的审批进度 |
| 流程自动恢复 | 全部高风险条款处理完成后，工作流自动 resume，进入报告生成 |

### 1.3 阶段边界

- **本文档不涉及**：文件上传（Teammate 1）、AI 扫描与分级路由（Teammate 2）
- **本文档涵盖**：双栏视图、三选项操作、防偏差设计、异步恢复、报告触发与导出、状态流转

## 二、HITL 视图布局规范

### 2.1 整体页面结构

HITL 审批页面采用三区域布局：

```
┌─────────────────────────────────────────────────────────────────┐
│  [页头]  合同标题 | 会话状态标签 | 审批进度条 | 退出保存按钮      │
├────────────────────────────┬────────────────────────────────────┤
│        [左栏]              │             [右栏]                  │
│  待审批条款列表 + 操作区    │     合同原文阅读区                   │
│  （固定宽度 42%）          │     （自适应宽度 58%）              │
│                            │                                    │
│  [条款卡片]                │  [原文段落，带高亮锚定]             │
│  风险等级 | 置信度 | 来源   │                                    │
│  AI 风险描述               │                                    │
│  ─────────────────         │                                    │
│  [Approve/Edit/Reject 区]  │                                    │
│                            │                                    │
├────────────────────────────┴────────────────────────────────────┤
│  [页脚]  当前条款 X/N | 跳转导航 | 已完成 / 待处理 计数          │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 左栏：风险条款摘要区规范

#### 2.2.1 条款卡片内容结构

每条待审批条款以卡片形式展示，卡片内容从上至下依次为：

| 区域 | 内容 | 视觉规范 |
|------|------|----------|
| **条款标题行** | 条款序号（如 #3）+ 条款类型标签（如"违约金不对等"） | 序号 16px 粗体；标签为圆角 badge |
| **风险等级行** | 风险等级标识（高/中/低）+ 色块 | 高风险：红色 `#D32F2F`；中风险：橙色 `#F57C00`；低风险：绿色 `#388E3C` |
| **置信度行** | 置信度数值（0–100）+ 进度条形式辅助展示 | 低于 60 时附加"低置信度"警示文字 |
| **判断来源行** | `规则触发` 或 `AI 推理` 标签，二者样式必须可区分 | 规则触发：蓝色实线边框 badge；AI 推理：紫色虚线边框 badge |
| **AI 风险描述** | `ReviewItem.ai_finding` 的完整文本内容 | 14px 常规字重，最多展示 6 行，超出折叠 |
| **原文摘要锚点** | 原文段落前 60 字符的预览 + "查看原文"链接 | 灰色斜体；点击触发右栏滚动定位 |
| **操作区** | Approve / Edit / Reject 三个操作控件（详见第三章） | 与卡片主体之间有分隔线 |

#### 2.2.2 条款来源标注规范

- **规则触发**（`source = rule_engine`）：必须在卡片上显示触发的具体规则 ID 或规则名称（如"规则 R-045：违约金单边约定"），并提供规则详情展开入口
- **AI 推理**（`source = ai_inference`）：必须在卡片上显示"AI 推理"标签，并以 tooltip 形式说明"此判断基于语义推理，不对应具体规则条文，置信度供参考"
- 两种来源不得使用相同的视觉样式，界面上必须使任何审核人员能够一眼区分

### 2.3 右栏：合同原文阅读区规范

#### 2.3.1 原文展示要求

- 以分页或长滚动方式展示合同全文，保留原始段落格式
- 原文区域右上角显示当前定位锚点标识（如"第 3 页，第 4 段"）
- 原文区域提供搜索框，支持关键词定位

#### 2.3.2 高亮锚定规范

- 当左栏某条款卡片处于"激活"状态时，右栏对应原文段落必须：
  1. 自动滚动至可见区域（smooth scroll）
  2. 以色块背景高亮标记（高风险条款使用浅红色背景 `#FFEBEE`）
  3. 在段落左侧显示竖线标记，颜色与风险等级色块一致
  4. 若条款跨段落，则所有关联段落均高亮

- 高亮状态在以下情况下解除：
  1. 用户点击左栏另一条款卡片
  2. 用户在右栏空白处点击（非高亮段落区域）

#### 2.3.3 原文区域操作限制

- 右栏原文为**只读展示**，不提供编辑功能
- 原文区域提供"复制段落"按钮（每个高亮段落旁），方便审核人员引用原文至备注
- 不提供删除、标注涂鸦等修改原文的功能

### 2.4 双向关联交互规范

双向关联是指左栏条款卡片与右栏原文段落之间的相互触发高亮机制。

#### 2.4.1 左 → 右（条款卡片触发原文高亮）

| 触发动作 | 系统响应 |
|----------|----------|
| 用户点击左栏任意条款卡片（非操作控件区域） | 右栏滚动至该条款对应的 `clause_location` 锚点，并高亮显示 |
| 用户使用键盘上下键在条款卡片列表中导航 | 右栏同步滚动并更新高亮 |
| 页面初始加载，第一条待处理高风险条款默认激活 | 右栏自动定位至第一条高风险条款的原文位置 |

#### 2.4.2 右 → 左（原文段落触发条款卡片激活）

| 触发动作 | 系统响应 |
|----------|----------|
| 用户点击右栏任意已高亮的原文段落 | 左栏对应条款卡片滚动至可见区域，并激活（展开操作区） |
| 用户点击右栏原文中未被任何条款关联的段落 | 无反应（或显示"该段落无风险标注"的轻提示 toast） |

#### 2.4.3 同步状态规范

- 任意时刻，左右两栏的激活状态保持严格一对一对应：左栏激活条款 = 右栏高亮原文
- 双向切换不产生历史堆栈（不触发浏览器 back/forward）
- 状态同步在 100ms 内完成（视觉无延迟感知）

## 三、审批操作规范

### 3.1 待审批条款列表排序与导航规范

#### 3.1.1 默认排序规则

待审批条款列表遵循以下优先级排序，排序在页面加载时固定，不随用户操作动态重排：

1. **风险等级**（高 > 中 > 低），高风险条款强制置顶
2. 同等级内按 **置信度降序**（置信度越高，AI 判断越确定，优先处理）
3. 同等级同置信度按 **条款在原文中的出现顺序**（`clause_location` 升序）

#### 3.1.2 进度指示规范

页头进度条与页脚计数实时反映审批进度：

- **页头进度条**：`已处理条款数 / 高风险条款总数`，以百分比和步骤数双重展示（如"3/7 条已完成"）
- **页脚计数**：分别统计"待处理"、"已通过"、"已修改"、"已驳回"四类数量
- 每次提交操作后，上述计数立即更新（乐观更新，后台持久化异步确认）

#### 3.1.3 条款导航规范

- 列表左侧提供序号锚点，支持直接点击跳转至任意条款卡片
- 页脚提供"上一条"/"下一条"导航按钮，仅在高风险待处理条款之间跳转（跳过已处理条款）
- 提供"跳转到下一条未处理"的快捷入口（每次提交后自动弹出确认提示："已保存，是否跳转到下一条？"）
- 不提供"全部跳过"或"全选通过"的批量操作入口

#### 3.1.4 条款状态标识

每条条款卡片左侧以图标标识当前处理状态：

| 状态 | 图标 | 说明 |
|------|------|------|
| `pending` | 灰色圆圈 | 待处理 |
| `approve` | 绿色对勾 | 已确认通过 |
| `edit` | 蓝色铅笔 | 已修改 AI 判断 |
| `reject` | 红色叉号 | 已驳回（标记为误报） |

---

### 3.2 Approve（确认）操作规范

#### 3.2.1 操作前提条件（防 Automation Bias 强制检查）

高风险条款执行 Approve 操作前，系统必须验证以下条件均已满足，否则 Approve 按钮保持禁用状态：

| 检查项 | 验证逻辑 | 未满足时的提示 |
|--------|----------|----------------|
| 原文已展开 | 用户已点击过"查看原文"或右栏已滚动至对应段落（通过交互事件记录判断） | "请先在右侧阅读原文段落后再确认" |
| 原文展开时长 | 用户在对应原文段落停留时间 ≥ 5 秒（防止机械点击，非严格计时，以事件记录为准） | 不显示倒计时，仅在条件未满足时禁用按钮 |
| 接受原因已填写 | `human_note` 字段非空，且长度 ≥ 10 个字符 | 输入框边框变红，提示"请填写接受原因（至少 10 字）" |

**中低风险条款**不强制要求上述前提，Approve 按钮默认可用，但仍鼓励填写备注。

#### 3.2.2 Approve 操作步骤

```
步骤 1：审核人员阅读左栏 AI 风险描述
步骤 2：点击"查看原文"或直接在右栏阅读对应高亮段落
步骤 3：在操作区"接受原因"文本框填写理由（高风险必填）
步骤 4：点击 [确认通过] 按钮
步骤 5：系统弹出确认对话框（仅高风险条款，见 3.2.3）
步骤 6：确认后系统提交，条款状态更新为 approve
步骤 7：显示成功 toast，自动聚焦到下一条待处理条款
```

#### 3.2.3 高风险条款 Approve 确认对话框

高风险条款点击"确认通过"后，必须弹出确认对话框，不可跳过：

```
┌──────────────────────────────────────────┐
│  确认接受此高风险条款                      │
│                                          │
│  条款类型：违约金不对等                    │
│  风险等级：高风险                          │
│  您已填写的接受原因：[显示用户填写内容]     │
│                                          │
│  确认后，此条款将被标记为"已接受"。         │
│  此操作可在报告生成前撤销。                │
│                                          │
│  [取消]              [确认接受]           │
└──────────────────────────────────────────┘
```

#### 3.2.4 Approve 提交数据规范

提交时写入 `ReviewItem` 的字段：

| 字段 | 值 |
|------|----|
| `human_decision` | `approve` |
| `human_note` | 用户填写的接受原因文本 |
| `decided_by` | 当前登录用户 ID |
| `decided_at` | 服务端当前 UTC 时间戳 |
| `is_false_positive` | `false` |

---

### 3.3 Edit（修改）操作规范

#### 3.3.1 可编辑字段范围

Edit 操作允许审核人员修改 AI 的判断内容，但**只能修改 AI 输出的描述性内容，不能修改合同原文**。

| 字段 | 可编辑 | 说明 |
|------|--------|------|
| `risk_level` | 是 | 下拉选择：高 / 中 / 低，审核人员可调整风险等级 |
| `ai_finding`（修正版） | 是 | 富文本输入框，可对 AI 风险描述进行修正、补充或删减 |
| `human_note` | 是 | 填写修改原因（必填，≥ 10 字） |
| `clause_text` | 否 | 合同原文段落，只读，不可修改 |
| `clause_location` | 否 | 原文定位锚点，系统维护，不可修改 |
| `confidence_score` | 否 | AI 置信度，修改后系统不重算，保留原值 |
| `source` | 否 | 判断来源，不可修改 |

#### 3.3.2 Edit 操作步骤

```
步骤 1：审核人员点击 [修改判断] 按钮，操作区切换为编辑模式
步骤 2：在"风险等级"下拉中选择修正后的等级（可与原等级相同）
步骤 3：在"风险描述（修正）"文本框中编辑 AI 的风险描述
步骤 4：在"修改原因"文本框填写修改理由（必填）
步骤 5：点击 [保存修改] 按钮
步骤 6：系统提交，条款状态更新为 edit
步骤 7：显示成功 toast，卡片更新展示修正后的内容
```

#### 3.3.3 修改内容记录规范

- 系统保留修改前后的快照，用于审计追溯
- 卡片上以"已修改"标签和蓝色铅笔图标标识该条款已被人工编辑
- 报告中对 Edit 条款展示"AI 原始判断"与"人工修正内容"的对照，并标注修改时间和操作人

#### 3.3.4 保存逻辑

- 点击"保存修改"时前端执行本地校验（必填项不为空）
- 校验通过后立即提交，不支持草稿暂存（未保存的编辑内容在离开当前卡片时提示确认）
- 提交成功前按钮显示 loading 状态，防止重复提交

#### 3.3.5 Edit 提交数据规范

| 字段 | 值 |
|------|----|
| `human_decision` | `edit` |
| `human_note` | 用户填写的修改原因 |
| `decided_by` | 当前登录用户 ID |
| `decided_at` | 服务端当前 UTC 时间戳 |
| `is_false_positive` | `false` |
| `ai_finding`（修正版） | 用户编辑后的风险描述（存入专用字段，不覆盖原始 `ai_finding`） |
| `risk_level`（修正版） | 用户选择的修正后风险等级（存入专用字段，不覆盖原始 `risk_level`） |

---

### 3.4 Reject（驳回）操作规范

#### 3.4.1 Reject 的业务语义

Reject 表示审核人员认为 AI 的风险判断本身有误（即该条款并不构成风险，AI 报告为误报）。Reject 操作会将该条款标记为 `is_false_positive = true`，作为规则库反馈数据，供后续模型或规则优化使用。

Reject **不等同于**"接受这个风险条款"，而是"否定 AI 的判断，认为此处无风险"。

#### 3.4.2 Reject 操作步骤

```
步骤 1：审核人员点击 [驳回判断] 按钮，操作区切换为驳回模式
步骤 2：在"驳回原因"文本框填写为何认为 AI 判断有误（必填，≥ 10 字）
步骤 3：确认"标记为误报"复选框（默认勾选，用户可取消）
步骤 4：点击 [确认驳回] 按钮
步骤 5：系统弹出轻量确认提示（非模态，inline 展示）
步骤 6：确认后系统提交，条款状态更新为 reject
步骤 7：显示成功 toast，卡片以"已驳回"样式展示
```

#### 3.4.3 标记为误报的逻辑

- `is_false_positive = true` 仅在 Reject 操作时由用户勾选后设置
- 若用户取消勾选"标记为误报"，则 `is_false_positive = false`，表示驳回但不归类为误报（适用于"虽然不是风险，但也不认为 AI 有问题"的边界情形）
- 误报数据定期汇总，供系统管理员优化规则库，不在本文档涉及的交互链路内处理

#### 3.4.4 Reject 提交数据规范

| 字段 | 值 |
|------|----|
| `human_decision` | `reject` |
| `human_note` | 用户填写的驳回原因 |
| `decided_by` | 当前登录用户 ID |
| `decided_at` | 服务端当前 UTC 时间戳 |
| `is_false_positive` | 用户勾选结果（`true` 或 `false`） |

---

### 3.5 操作的可撤销规范

- 在报告生成触发前，已提交的 Approve / Edit / Reject 操作均可被撤销
- 撤销入口：条款卡片右上角"撤销"链接（仅在已处理状态下显示）
- 撤销后，`human_decision` 重置为 `pending`，相关字段清空，操作记录保留历史快照
- 报告生成流程触发后（`ReviewSession.state = completed`），不再允许撤销

## 四、防 Automation Bias 设计规范

### 4.1 设计原则

Automation Bias（自动化偏见）是指用户对自动化系统的输出过度信任，不加批判地接受 AI 判断。在合同审核场景中，这一风险尤为突出：审核人员可能因工作量大而机械点击"确认通过"，而不实际阅读 AI 判断与原文。

本系统通过以下设计手段，在不显著增加操作摩擦的前提下，强制审核人员真正阅读并独立判断。

### 4.2 高风险条款专项约束

以下约束仅对 `risk_level = high` 的条款生效，中低风险条款不适用：

| 约束项 | 实现方式 | 目的 |
|--------|----------|------|
| **强制展开原文** | 进入高风险条款卡片时，右栏自动滚动定位；Approve 按钮在用户触达原文对应区域前保持禁用 | 确保审核人员看到原文 |
| **必须填写接受原因** | `human_note` 非空且 ≥ 10 字才能提交 Approve；系统不提供预设原因供点选 | 要求独立形成判断 |
| **二次确认对话框** | Approve 提交前弹出确认对话框（详见 3.2.3）| 给审核人员一次反悔机会 |
| **禁止批量通过** | 不提供全选、批量 Approve 的入口或 API；后端也拒绝批量决策请求 | 防止机械操作 |
| **操作间隔限制** | 同一审核会话中，若连续 5 条高风险条款均在 10 秒内 Approve，系统弹出警示："您的操作速度较快，请确认已仔细阅读每条原文。" | 识别异常操作模式 |

### 4.3 AI 判断的不确定性展示规范

- 所有条款卡片的 AI 判断描述必须使用**非绝对化表述**，禁止"该条款构成违规"等确定性语言
- 推荐表述模板："AI 检测到该条款可能存在…风险，请结合原文判断"
- 置信度低于 60 时，在卡片顶部显示醒目警示条："此判断置信度较低，请务必仔细阅读原文后再决策"
- 所有 AI 推理来源的条款必须在 tooltip 中说明推理的局限性（见 2.2.2）

### 4.4 来源可区分性规范

- 界面上"规则触发"与"AI 推理"两种来源必须通过至少两种视觉维度区分（颜色 + 边框样式 / 图标形状），不可仅依赖颜色（色盲友好）
- 规则触发的条款，审核人员可展开查看触发的具体规则条文，帮助其独立验证判断
- AI 推理的条款，审核人员可展开查看 AI 分析的关键片段和推理路径（在系统能力范围内）

## 五、异步中断与恢复规范

### 5.1 中断场景分类

| 中断类型 | 触发条件 | 系统处理方式 |
|----------|----------|--------------|
| **主动退出** | 用户点击"退出"/"保存并离开"按钮 | 立即持久化当前进度，返回合同列表 |
| **浏览器关闭/刷新** | 浏览器 beforeunload 事件 | 显示离开提示，当前进度已自动保存 |
| **会话超时** | 用户长时间无操作（默认 30 分钟） | 自动保存并弹出"会话即将过期"提示 |
| **网络中断** | API 请求失败 | 本地缓存操作，恢复连接后自动同步 |

### 5.2 状态保存规范

#### 5.2.1 自动保存触发时机

- 每次 Approve / Edit / Reject 提交成功后立即持久化（通过 LangGraph Checkpointer）
- 用户在接受原因、修改原因等文本框输入内容时，每 30 秒自动保存草稿（仅前端本地 Storage）
- 用户点击"保存并离开"时立即同步持久化

#### 5.2.2 持久化内容范围

通过 LangGraph Checkpointer 持久化的内容包括：

- `ReviewSession.state`（当前会话状态）
- 所有 `ReviewItem.human_decision` 及相关字段（已完成的决策）
- 当前审批进度（已处理条款数、最后处理的条款 ID）
- LangGraph 工作流的中间状态（thread_id 对应的 checkpoint）

草稿内容（未提交的文本框内容）仅保存在浏览器本地 Storage，不持久化到服务端。

#### 5.2.3 中途离开时的提示规范

当系统检测到用户即将离开（beforeunload 或点击退出）时：

```
[情况 A：有未提交的文本框内容]
弹出提示框（阻止离开）：
  "您有未保存的内容（当前正在编辑的条款）。
   离开将丢失该条款的未提交内容，已完成的其他条款已自动保存。
   [继续离开]   [返回继续审批]"

[情况 B：无未提交内容，所有已处理条款均已持久化]
不弹出提示，直接离开。
页面在跳转前显示轻量 toast："审批进度已保存"
```

### 5.3 断点恢复交互规范

#### 5.3.1 恢复入口

用户重新进入系统时，在以下位置提供恢复入口：

1. **合同列表页**：状态为 `hitl_pending` 的合同显示"待继续审批"标签和"继续审批"按钮
2. **首页仪表盘**：显示"待我处理"模块，列出所有 `hitl_pending` 的会话，每条显示进度百分比
3. **邮件/通知**（如系统集成了通知功能）：包含直接跳转到断点的深度链接

#### 5.3.2 恢复后的页面状态

用户点击"继续审批"进入审批页面时：

- 页面自动加载该会话的最新 Checkpoint 状态
- 已处理的条款卡片显示对应的处理结果（Approve / Edit / Reject 状态）
- 列表自动滚动至**第一条 `human_decision = pending` 的高风险条款**（即断点位置）
- 右栏同步定位至该条款的原文段落
- 页头显示恢复提示 banner（5 秒后自动消失）：

```
┌─────────────────────────────────────────────────────────┐
│  已恢复上次审批进度（上次保存时间：2026-03-10 17:32）      │
│  当前进度：已完成 4/7 条，继续从第 5 条开始              │   [x]
└─────────────────────────────────────────────────────────┘
```

#### 5.3.3 跨天恢复场景（LangGraph Checkpointer 支撑）

LangGraph Checkpointer 持久化机制支持工作流状态的跨会话保存，使跨天恢复成为可能：

- `ReviewSession.langgraph_thread_id` 是 LangGraph Checkpointer 的线程标识，与审批会话一对一绑定
- 工作流在 `interrupt` 节点等待时，完整的图状态（包括已处理条款的决策、待处理条款队列）均持久化在 Checkpointer 后端
- 用户第二天（或任意时间后）重新打开系统，系统通过 `thread_id` 恢复 LangGraph 状态，前端展示与中断前完全一致
- 跨天场景下，恢复提示 banner 中的"上次保存时间"会清晰显示跨天信息，如"上次保存时间：昨天 17:32"

#### 5.3.4 并发访问保护

- 同一审批会话在同一时间只允许一个活跃用户操作（防止多人同时操作同一 `ReviewSession`）
- 若检测到同一会话已有另一用户打开，显示只读视图并提示："该合同正由 [用户名] 审批中，您可以查看进度，但暂时无法操作"
- 锁的有效期：活跃用户最后一次操作后 5 分钟内保持锁定；超时后自动释放

## 六、报告生成与导出规范

### 6.1 流程自动 Resume 触发条件

以下条件同时满足时，系统自动触发 LangGraph 工作流的 `resume`，进入报告生成阶段：

- 该 `ReviewSession` 中所有 `risk_level = high` 的 `ReviewItem.human_decision` 均不为 `pending`
- 当前用户点击了最后一条高风险条款的提交操作（系统在提交成功回调中检查上述条件）

触发顺序：

```
最后一条高风险条款提交成功
  → 后端检查：所有高风险条款 human_decision != pending ✓
  → LangGraph resume() 调用
  → ReviewSession.state = completed
  → 异步启动报告生成任务
  → 前端收到状态变更通知
  → 显示"审批完成，报告生成中"页面
```

### 6.2 报告生成期间的用户界面

报告生成为异步任务，期间展示等待页面：

```
┌──────────────────────────────────────────────────────────┐
│  合同审核已完成，正在生成报告…                              │
│                                                          │
│  [动态进度指示器]                                         │
│                                                          │
│  审批汇总：                                               │
│  · 已确认（Approve）：3 条                               │
│  · 已修改（Edit）：2 条                                  │
│  · 已驳回（Reject）：2 条                                │
│  · 自动通过（低风险）：12 条                             │
│                                                          │
│  报告通常在 30 秒内生成完成，请稍候。                     │
│  您也可以 [返回合同列表] ，报告生成后会通知您。            │
└──────────────────────────────────────────────────────────┘
```

### 6.3 审批完成状态页面规范

报告生成完成后（`ReviewSession.state = report_ready`），页面切换至审批完成状态：

```
┌──────────────────────────────────────────────────────────┐
│  审核报告已生成                                            │
│  合同名称：[合同标题]                                     │
│  审核完成时间：2026-03-11 14:23                           │
│  审核人员：[用户名]                                        │
│                                                          │
│  整体风险结论：[高风险 / 中等风险 / 低风险]               │
│  [条款汇总统计图]                                         │
│                                                          │
│  ┌───────────────────────────────────────┐              │
│  │   下载审核报告                          │              │
│  │   [下载 PDF 报告]  [下载 JSON 数据]    │              │
│  └───────────────────────────────────────┘              │
│                                                          │
│  [查看完整报告在线预览]   [返回合同列表]                   │
└──────────────────────────────────────────────────────────┘
```

### 6.4 报告内容规范

#### 6.4.1 报告必须包含的内容

| 内容块 | 具体内容 | 来源字段 |
|--------|----------|----------|
| **执行摘要** | 合同基本信息（主体、金额、日期）、整体风险结论、审核完成时间 | `ExtractedField` + `ReviewSession` |
| **条款审核详情** | 每条 `ReviewItem` 的：风险等级、AI 判断（含原始 `ai_finding`）、人工决策（`human_decision`）、操作人、操作时间、人工备注 | `ReviewItem` 全字段 |
| **Edit 条款的前后对照** | 修改前 AI 判断 vs 修改后人工判断，按条款逐一展示 | `ReviewItem` 历史快照 |
| **覆盖范围声明** | 明确列出本次审核覆盖的条款类型（如：违约金条款、保密条款、知识产权条款）和明确未覆盖的条款类型 | `ReviewReport.coverage_statement` |
| **免责声明** | "本报告由 AI 辅助分析生成，所有最终判断由人工审核人员 [用户名] 负责。AI 分析结果仅供参考，不构成法律意见。" | 系统模板，固定文本 |
| **操作审计日志** | 每步操作的时间戳、操作人、操作类型（提交 / 撤销 / 修改）的完整记录 | Checkpointer 持久化日志 |

#### 6.4.2 报告禁止包含的内容

- 不得使用"审核通过"、"合规"、"无风险"等绝对化结论
- 不得省略覆盖范围声明或免责声明
- 不得遗漏任何 `human_decision != pending` 的条款记录

### 6.5 报告下载交互规范

#### 6.5.1 PDF 报告

- 格式：标准 A4 页面，包含页眉（合同标题）、页脚（页码 + 生成时间 + 免责声明）
- 文件名规则：`{合同标题}_{审核完成日期}_审核报告.pdf`（如 `采购合同_2026-03-11_审核报告.pdf`）
- 点击"下载 PDF 报告"后立即开始下载（如文件较大，显示生成中 loading）
- 已生成的 PDF 缓存服务端 24 小时，重复下载不重新生成

#### 6.5.2 JSON 数据导出

- 格式：UTF-8 编码 JSON，结构对应 `ReviewReport.detail_json`
- 文件名规则：`{合同标题}_{审核完成日期}_审核数据.json`
- JSON 结构包含所有 `ReviewItem` 的完整字段（含 `decided_by`、`decided_at`、`is_false_positive`）
- 用途说明：供下游系统集成，字段定义参见 `08_api_spec`

#### 6.5.3 在线预览

- 提供报告的在线 HTML 预览页面，支持打印
- 在线预览与 PDF 内容一致，但支持条款内链跳转（点击条款可跳转至对应原文段落）
- 在线预览页面为只读，不支持任何编辑操作

## 七、状态流转定义

### 7.1 ReviewItem.human_decision 枚举定义

| 枚举值 | 含义 | 可转入的前置状态 | 转入条件 |
|--------|------|-----------------|----------|
| `pending` | 待处理，审核人员尚未做出决策 | 初始状态（系统创建时默认） | 系统初始化 ReviewItem 时设置；或审核人员撤销已有决策后重置 |
| `approve` | 已确认，审核人员接受 AI 的风险判断 | `pending` | 用户完成 Approve 操作流程，`human_note` 非空（高风险条款），服务端写入 |
| `edit` | 已修改，审核人员修改了 AI 的风险等级或描述 | `pending` | 用户完成 Edit 操作流程，`human_note` 非空，服务端写入 |
| `reject` | 已驳回，审核人员认为 AI 判断有误 | `pending` | 用户完成 Reject 操作流程，`human_note` 非空，服务端写入 |

### 7.2 状态流转图

```
                        ┌─────────────┐
                        │   pending   │  ← 初始状态 / 撤销后回退
                        └──────┬──────┘
                               │
             ┌─────────────────┼─────────────────┐
             │                 │                 │
             ▼                 ▼                 ▼
        ┌─────────┐       ┌─────────┐       ┌─────────┐
        │ approve │       │  edit   │       │ reject  │
        └────┬────┘       └────┬────┘       └────┬────┘
             │                 │                 │
             └────────────┬────┘                 │
                          │                      │
                          ▼                      │
                    [可撤销状态]  ─── 撤销 ──→ pending
                          │
                          │ 报告生成后不可撤销
                          ▼
                    [最终状态：锁定]
```

### 7.3 状态转换约束

| 转换 | 允许条件 | 禁止情况 |
|------|----------|----------|
| `pending → approve` | 用户完成 Approve 操作流程，高风险条款必须有非空 `human_note` | 无 `human_note`（高风险）；`ReviewSession.state = completed` 后 |
| `pending → edit` | 用户完成 Edit 操作流程，`human_note` 非空 | `ReviewSession.state = completed` 后 |
| `pending → reject` | 用户完成 Reject 操作流程，`human_note` 非空 | `ReviewSession.state = completed` 后 |
| `approve/edit/reject → pending` | 用户主动撤销操作 | `ReviewSession.state = completed` 后（报告已生成） |
| 批量状态更新 | — | **严禁**，每条 ReviewItem 必须单独提交决策 |

### 7.4 ReviewSession.state 与 ReviewItem.human_decision 的联动

| ReviewSession.state | 对 ReviewItem 的限制 |
|---------------------|---------------------|
| `hitl_pending` | 所有操作（Approve / Edit / Reject / 撤销）均开放 |
| `completed` | 所有 ReviewItem 的 `human_decision` 进入只读状态，不可再修改或撤销 |
| `aborted` | 所有 ReviewItem 的操作入口关闭，页面仅展示最后一次的状态快照 |

### 7.5 ReviewItem.human_decision 与报告生成的关联规则

- 报告生成触发条件：`ReviewSession` 中所有 `risk_level = high` 的 `ReviewItem.human_decision ∈ {approve, edit, reject}`（即无 `pending`）
- 中低风险条款（`risk_level ∈ {medium, low}`）的 `human_decision` 不影响报告生成触发
- 中低风险条款的 `human_decision` 若仍为 `pending`（未处理），在报告中以"AI 自动通过（人工未复核）"标注
- 报告中所有条款记录，无论 `human_decision` 值为何，均完整呈现，不可省略
