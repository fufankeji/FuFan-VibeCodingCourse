**文档编号**：04_interaction_design/t2_review_states
**编写日期**：2026-03-11
**负责范围**：Teammate 2（中间状态处理）
**输入文档**：`03_problem_modeling/problem_model.md`
**衔接文档**：
- 上游：`04_interaction_design/t1_upload_parsing.md`（文件上传与解析，Teammate 1 负责）
- 下游：`04_interaction_design/t3_hitl_decision.md`（HITL 审批操作，Teammate 3 负责）

## 一、阶段概述

本文档描述合同审核系统中从"解析完成"到"进入 HITL 等待"的完整中间阶段交互链路，具体涵盖以下状态段：

```
[解析完成] → 结构化字段呈现 → AI 风险扫描（scanning）→ 分级路由判断
    ↓                                                          ↓
[低风险：自动通过]   [中风险：进入复核队列]   [高风险：触发 interrupt → hitl_pending]
    ↓
[失败状态：系统级 / 业务级]
```

**核心设计原则**（来自 problem_model.md 第五章设计红线）：

1. 所有 AI 判断必须附带置信度标识，禁止绝对化表述
2. 界面必须明确区分"规则触发"与"AI 推理"两种判断来源
3. 操作全程可追溯，每步状态变更均通过 Checkpointer 持久化
4. 用户在任何等待状态下都必须有清晰的进度感知和预期管理

## 二、各中间状态的界面规范

### 2.1 状态：结构化字段呈现（post-parsing）

#### 触发条件

- ReviewSession.state 由 `parsing` 转为 `scanning` 前的过渡展示阶段
- OCR 与结构化提取完成，ExtractedField 记录已落库
- 由 Teammate 1 的解析流程结束后自动触发本视图

#### 界面呈现

**布局结构**：双栏布局

- 左栏（60%宽）：结构化字段卡片组
- 右栏（40%宽）：合同原文预览（PDF 渲染/文本渲染），支持与字段联动跳转

**字段卡片组规范**：

每个 ExtractedField 渲染为一张信息卡片，包含以下元素：

| 元素 | 说明 |
|------|------|
| 字段标签 | 中文名称（如"合同主体（甲方）"、"合同金额"） |
| 字段值 | AI 提取的文本值，字号稍大作为主要内容 |
| 置信度指示器 | 进度条样式，0-100%，配以数字，**不得隐藏此元素** |
| 置信度分级标签 | 高（≥85%）/ 中（60-84%）/ 低（<60%），用色彩区分 |
| 来源锚点按钮 | 点击后右栏原文滚动至对应段落并高亮显示 |
| 核对状态徽标 | 未核对 / 已确认 / 已修改，初始状态为"未核对" |

**置信度色彩规范**：

- 高置信度（≥85%）：绿色标签，正常展示
- 中置信度（60-84%）：黄色标签，无特殊提示
- 低置信度（<60%）：橙红色标签，必须展示核对引导提示

#### 低置信度字段的标记与核对引导

当 `ExtractedField.needs_human_verification = true`（即置信度 < 60%）时，触发以下交互：

**字段卡片状态变化**：
- 卡片左侧出现橙色竖线边框
- 卡片顶部显示醒目提示条，文案："AI 提取置信度较低，请人工核对原文确认"
- 提示条内嵌"跳转原文"按钮，点击后右栏原文定位至对应字段的来源段落

**核对操作流程**：
1. 用户点击"确认字段值"按钮：弹出行内编辑框，用户可修改或确认当前值，点击保存后核对状态更新为"已确认"或"已修改"
2. 用户直接点击"我已核对，值无误"：核对状态更新为"已确认"，置信度提示收起
3. 如用户未对低置信度字段执行任何操作，直接进入下一步时：系统弹出确认对话框，列出所有未核对的低置信度字段，要求用户确认是否继续（允许跳过，但记录"用户选择跳过核对"事件）

**页面级状态提示**：

当存在低置信度字段时，页面顶部显示橙色信息横幅：
> "发现 N 个字段提取置信度较低，建议人工核对后再开始风险扫描"

横幅右侧提供"查看全部低置信度字段"跳转锚点和"忽略，直接开始扫描"操作按钮。

#### 用户可见信息

- 本次解析共提取了 N 个结构化字段
- 高置信度字段数 / 低置信度字段数（数字摘要）
- 每个字段的具体值与置信度
- 每个字段在原文中的定位位置
- 操作指引：核对完成后点击"开始 AI 风险扫描"进入下一阶段

#### 系统后台行为

- ExtractedField 记录已持久化，此阶段用户的核对/修改操作实时更新字段记录
- ReviewSession.state 保持在 `scanning`（准备就绪标志位），等待用户触发风险扫描
- 用户的核对操作（确认/修改/跳过）记录到操作日志，供审计追溯

---

### 2.2 状态：AI 风险扫描进行中（scanning）

#### 触发条件

- 用户点击"开始 AI 风险扫描"或系统在字段核对完成后自动触发
- ReviewSession.state = `scanning`
- LangGraph 工作流在 `risk_scanning` 节点执行中

#### 界面呈现

**页面主体状态**：扫描进行中视图，替换字段展示视图

**扫描进度展示组件**（核心组件，不可省略）：

```
┌─────────────────────────────────────────────────────┐
│  AI 风险扫描进行中                                    │
│  ─────────────────────────────────────────────────  │
│  [●●●●●●●○○○○○○○○○○○○○]  32%                        │
│                                                     │
│  正在分析：第 3 / 9 章节  · 已发现 2 处潜在风险       │
│                                                     │
│  预计剩余时间：约 45 秒                               │
│  ─────────────────────────────────────────────────  │
│  扫描内容：单边条款 ✓  · 违约金条款 ✓  · 兜底条款 …  │
└─────────────────────────────────────────────────────┘
```

**进度反馈规范**：

| 反馈元素 | 说明 |
|----------|------|
| 整体进度条 | 基于章节/段落数量估算，0-100%；若无法获取精确进度，显示不确定性动画（indeterminate） |
| 当前处理章节 | "正在分析第 X / N 章节"，章节信息来自文档结构解析 |
| 已发现风险计数 | 实时更新的数字，降低用户焦虑，增加扫描"实感" |
| 扫描维度清单 | 列出本次扫描覆盖的条款类型（单边条款、违约金不对等、弹性表述等），已完成的打勾，进行中的显示动画 |
| 预计剩余时间 | 基于平均处理速度估算，标注"约"以体现不确定性 |

**重要展示约束**：

- 扫描进行中禁止显示任何"扫描完成"、"无风险"等结论性文字
- 已发现的风险条目数量可以实时显示，但不显示具体内容（避免用户在扫描未完成时过早聚焦某条款）
- 扫描维度清单必须列出，体现"覆盖范围"（对应设计红线第三条）

**等待期间可用操作**：

- 查看已核对的结构化字段（只读模式，可切换视图）
- 查看合同原文（只读）
- 不提供"取消扫描"按钮（避免用户轻易中断，导致数据不完整）；可提供"后台运行，先去处理其他事务"功能，允许用户离开此页面，扫描完成后推送通知

#### 用户可见信息

- 当前处于"AI 初筛"阶段的位置说明（工作流进度指示器中高亮此节点）
- 扫描进度百分比与文字描述
- 已发现风险数量（数字，不含详情）
- 扫描预计完成时间
- 扫描维度覆盖说明（增强信任感）

#### 系统后台行为

- LangGraph 工作流在 `risk_scanning` 节点调用 LLM + 规则引擎，对合同全文执行风险扫描
- 每发现一条风险项，实时创建 ReviewItem 记录（持久化）
- 扫描进度通过 WebSocket 或 SSE 推送至前端，保证实时性
- ReviewSession.state 维持在 `scanning`
- 全部扫描完成后，LangGraph 工作流根据 ReviewItem 风险等级分布执行路由判断

---

### 2.3 状态：风险扫描完成 — 路由判断

#### 触发条件

- LangGraph `risk_scanning` 节点完成，进入路由判断节点
- 路由判断基于 ReviewItem 中 `risk_level = high` 的条款数量

#### 路由逻辑（界面不显示此逻辑，但需理解以设计对应响应）

| 条件 | 路由结果 | ReviewSession.state 目标值 |
|------|----------|--------------------------|
| 无高风险条款 + 低风险条款数量在阈值内 | 自动通过 | `completed`（直接跳过 HITL） |
| 存在中风险条款，无高风险 | 进入批量复核队列 | `hitl_pending`（type: batch_review） |
| 存在至少一条高风险条款 | 触发 LangGraph interrupt | `hitl_pending`（type: interrupt） |

**注意**：路由结果的状态过渡在系统后台完成，界面感知到的是目标状态的展示，详见第三章。

## 三、三档分级路由的交互差异规范

### 3.1 低风险路由：自动通过

#### 用户感知方式

扫描完成后，界面平滑过渡至"扫描结果摘要"视图，显示：

```
┌─────────────────────────────────────────────────────┐
│  ✓  AI 风险扫描完成                                   │
│  ─────────────────────────────────────────────────  │
│  综合风险评估：低风险                                  │
│  置信度：87%  ·  来源：规则引擎 + AI 推理              │
│                                                     │
│  本次扫描共识别 12 处条款，均属低风险范围              │
│  流程已自动推进至报告生成阶段                          │
│                                                     │
│  ⚠ 注：AI 辅助评估，最终合规责任由审核人员承担         │
└─────────────────────────────────────────────────────┘
```

**关键设计要点**：

- "低风险"标签必须附带置信度数字，不得仅显示颜色
- 必须显示免责声明（对应设计红线第二条）
- 自动推进时不弹出打断性弹窗，仅以页面平滑动画过渡
- 审核人员仍可点击"查看详细扫描结果"查阅所有低风险条款（只读浏览，不需要操作）

**可用操作**：

| 操作 | 说明 |
|------|------|
| 查看详细扫描结果 | 展开所有 ReviewItem，可按来源（规则/AI）筛选 |
| 查看合同原文 | 支持条款高亮与原文联动 |
| 等待报告生成 | 系统自动进行，无需用户操作 |

**不提供的操作**：

- 不提供"撤回自动通过"按钮（低风险无需干预，避免界面混乱）
- 不提供批量修改风险等级的入口（需联系管理员调整阈值）

---

### 3.2 中风险路由：进入批量复核队列

#### 触发条件

- 风险扫描结果中存在 `risk_level = medium` 的条款，且无 `risk_level = high` 的条款
- ReviewSession.state = `hitl_pending`，type 标记为 `batch_review`

#### 界面响应

扫描完成后，过渡至"中风险复核视图"：

```
┌─────────────────────────────────────────────────────┐
│  需要人工复核                                         │
│  ─────────────────────────────────────────────────  │
│  发现 5 条中风险条款，需要您的确认                     │
│  预计处理时间：约 10 分钟                             │
│                                                     │
│  您可以逐条查看，也可以批量确认未修改项                │
│  [开始逐条复核]  [批量复核模式]                        │
└─────────────────────────────────────────────────────┘
```

**复核列表视图规范**：

| 列元素 | 说明 |
|--------|------|
| 条款序号 | 列表中的序号，非合同原始编号 |
| 风险标签 | 中风险（黄色），配置信度数字 |
| 来源标签 | "规则触发" 或 "AI 推理"（**必须显示，不可省略**） |
| 风险描述摘要 | AI 或规则引擎生成的风险说明（单行截断，可展开） |
| 原文片段 | 对应合同原文的前 50 字，配"查看完整原文"链接 |
| 操作按钮 | 确认 / 修改描述 / 标记误报 |
| 处理状态徽标 | 待处理 / 已确认 / 已修改 / 已标记误报 |

**批量操作规范**：

- 中风险条款允许批量勾选后一键"批量确认"
- 批量确认前弹出摘要确认对话框，展示将被批量确认的条款列表，要求用户点击"确认全部"
- 批量确认后，所有被确认的 ReviewItem.human_decision 更新为 `approve`，decided_by 和 decided_at 记录当前用户与时间

**来源标签展示约束**：

- "规则触发"来源：使用图标 + 蓝色标签，Tooltip 内容："此风险由预设规则库匹配触发"
- "AI 推理"来源：使用图标 + 紫色标签，Tooltip 内容："此风险由 AI 模型推断，存在不确定性，请结合原文判断"

**全部处理完成后的状态变化**：

所有中风险条款处理完成后，页面显示完成提示，ReviewSession.state 自动流转至 `completed`，进入报告生成阶段。

---

### 3.3 高风险路由：触发 LangGraph interrupt

#### 触发条件

- 风险扫描结果中存在至少一条 `risk_level = high` 的条款
- LangGraph 在 `routing` 节点调用 `interrupt()`，工作流暂停
- ReviewSession.state = `hitl_pending`，type 标记为 `interrupt`

#### 界面响应：流程暂停提示

扫描完成后，不自动过渡，而是以明显的状态中断视觉设计告知用户：

```
┌═════════════════════════════════════════════════════╗
║  ! 流程已暂停 — 需要您处理高风险条款                  ║
║  ─────────────────────────────────────────────────  ║
║  发现 3 条高风险条款                                  ║
║  置信度范围：72% - 91%                               ║
║                                                     ║
║  ⛔ 高风险条款不允许批量通过，需逐条展开确认            ║
║                                                     ║
║  这些条款可能涉及重大法律义务或不对等风险，             ║
║  AI 判断仅供参考，最终决策由您负责                     ║
║                                                     ║
║  [立即处理高风险条款] ←主操作入口，跳转至 Teammate 3 范围║
║                                                     ║
║  预计处理时间：约 15-30 分钟（取决于条款复杂度）        ║
╚═════════════════════════════════════════════════════╝
```

**视觉设计规范**：

- 边框使用橙红色双线或粗线，与低/中风险视图有明显区分
- 标题前缀使用感叹号图标（非错误图标，避免误解为系统错误）
- "流程已暂停"措辞准确反映 LangGraph interrupt 的语义，避免使用"审核失败"等负面措辞
- 禁止在此视图提供任何"跳过高风险条款"或"一键通过"的操作入口

**工作流暂停期间的系统状态**：

- LangGraph checkpoint 已持久化当前工作流状态，支持跨天异步恢复
- ReviewSession.state = `hitl_pending` 持续直到所有高风险条款处理完成
- 页面刷新后状态保留，用户可从任意设备恢复操作（通过 langgraph_thread_id 关联）

**并存的中风险条款处理**：

如同时存在高风险和中风险条款，显示优先级说明：
> "发现 3 条高风险条款和 2 条中风险条款。请先逐条处理高风险条款，中风险条款可在完成后批量复核。"

高风险和中风险条款在列表中分组显示，高风险组置顶，且禁止跳过直接操作中风险条款。

## 四、LangGraph 工作流状态可视化规范

### 4.1 可视化组件位置与形式

**位置**：页面顶部固定区域（全局导航栏下方），高度 64px，在所有审核阶段页面持续显示

**形式**：水平线性步骤条（Stepper），节点间用连线连接

**展示内容**：

```
[上传解析] ──✓── [字段核对] ──✓── [AI 扫描] ──●── [分级路由] ──○── [人工审核] ──○── [报告生成]
```

### 4.2 节点状态定义与视觉规范

| 节点状态 | 视觉表现 | 说明 |
|----------|----------|------|
| 待处理（pending） | 空心圆圈，灰色线条，灰色文字 | 尚未开始的步骤 |
| 进行中（active） | 实心圆圈 + 脉冲动画，蓝色高亮，加粗文字 | 当前正在执行的节点 |
| 已完成（completed） | 勾选图标，绿色实心，正常文字 | 已成功通过的节点 |
| 暂停中（interrupted） | 感叹号图标，橙色实心，橙色文字 | LangGraph interrupt 触发，等待人工介入 |
| 失败（failed） | 叉号图标，红色，红色文字 | 系统级或业务级失败 |
| 跳过（skipped） | 虚线圆圈，灰色，斜体文字 | 如低风险路由跳过 HITL 节点 |

### 4.3 节点详情 Tooltip

鼠标悬停（或移动端点击）任意节点时，显示 Tooltip，内容包含：

| 信息项 | 说明 |
|--------|------|
| 节点名称 | 中文名称（与 LangGraph node 名称映射） |
| 状态文字 | 对应上表状态的文字描述 |
| 时间信息 | 已完成节点显示"完成于 YYYY-MM-DD HH:mm"；进行中显示"开始于 HH:mm"；暂停中显示"暂停于 HH:mm" |
| 操作提示 | 进行中节点显示当前操作说明；暂停节点显示"需要您的操作"入口链接 |

### 4.4 工作流节点与 ReviewSession.state 的映射关系

| 步骤条节点名称 | 对应 LangGraph 节点 | 对应 ReviewSession.state | 备注 |
|----------------|---------------------|--------------------------|------|
| 上传解析 | `file_ingestion` | `parsing` | Teammate 1 范围 |
| 字段核对 | `field_verification` | `parsing`（扩展状态） | 用户交互阶段，state 不变 |
| AI 扫描 | `risk_scanning` | `scanning` | 本文档核心 |
| 分级路由 | `routing` | `scanning`（路由判断中） | 路由后立即转入目标 state |
| 人工审核 | `hitl_review` | `hitl_pending` | Teammate 3 范围 |
| 报告生成 | `report_generation` | `completed` | 触发报告生成任务 |

### 4.5 可视化刷新机制

- 前端通过 WebSocket 或 Server-Sent Events 订阅 ReviewSession.state 变更事件
- 收到状态变更事件后，步骤条对应节点状态即时更新，无需整页刷新
- 用户离开页面后重新打开，前端通过 API 查询当前 ReviewSession 的状态，恢复正确的步骤条显示

## 五、失败状态处理规范

### 5.1 失败类型定义

本系统定义两类失败，在界面呈现和处理路径上有明确区分：

| 类型 | 定义 | 示例 |
|------|------|------|
| **系统级失败** | 技术组件异常导致流程无法继续 | OCR 服务超时、LLM API 调用失败、网络断开 |
| **业务级失败** | 合同文件本身存在导致流程无法继续的问题 | 合同文件损坏、扫描件清晰度过低、内容无法识别（加密/乱码） |

### 5.2 系统级失败

#### 触发条件

- OCR 服务调用超时（超过配置的超时阈值，如 60 秒）
- LLM API 返回非 2xx 状态码或响应解析失败
- 内部服务调用错误（5xx 错误）
- LangGraph 工作流执行中抛出未捕获异常

#### 界面呈现

```
┌─────────────────────────────────────────────────────┐
│  系统处理遇到问题                                     │
│  ─────────────────────────────────────────────────  │
│  在 [AI 风险扫描] 阶段遇到技术问题，处理已暂停         │
│                                                     │
│  错误类型：AI 分析服务响应超时                         │
│  发生时间：2026-03-11 14:32:07                        │
│  错误参考码：ERR-20260311-A8F2（用于联系支持时查询）   │
│                                                     │
│  ─────────────────────────────────────────────────  │
│  您可以尝试：                                         │
│  [重试此步骤]  [联系技术支持]  [提交问题反馈]          │
│                                                     │
│  注：合同文件和已完成的处理步骤均已保存，重试后可继续    │
└─────────────────────────────────────────────────────┘
```

**关键设计要点**：

- 错误信息使用用户可理解的描述，不暴露技术栈细节（如不显示"500 Internal Server Error"或堆栈信息）
- 显示错误参考码，便于用户联系支持时提供上下文
- 明确告知用户已完成的工作已保存，消除用户担忧"是否需要重新上传"
- 步骤条中对应节点更新为"失败"状态

#### 重试机制规范

- "重试此步骤"操作仅重试失败的节点，不重新执行已完成的节点
- 重试前检查系统健康状态，若已知服务仍不可用，则提示"当前服务暂时不可用，建议稍后重试"而非直接重试
- 最多允许用户手动重试 3 次；超过 3 次提示联系支持并禁用重试按钮
- 重试操作记录到操作日志（含重试次数、时间戳）

---

### 5.3 业务级失败

#### 触发条件

- 合同文件经 OCR 处理后可提取文本量低于最低阈值（如有效文本字数 < 200）
- 文件加密或设有密码保护，无法读取内容
- 文件内容经校验确认为非合同类文档（如图片集、空白文档）
- 合同文本语言不在系统支持范围内

#### 界面呈现

```
┌─────────────────────────────────────────────────────┐
│  无法处理此合同文件                                   │
│  ─────────────────────────────────────────────────  │
│  问题类型：合同内容无法有效识别                        │
│                                                     │
│  具体原因：上传文件的可识别文本量不足，可能原因：       │
│  · 扫描件清晰度过低（建议 DPI ≥ 300）                │
│  · 文件为纯图片格式，未包含可识别文字                  │
│  · 文件内容被加密保护                                 │
│                                                     │
│  ─────────────────────────────────────────────────  │
│  建议操作：                                           │
│  [重新上传合同文件]  [联系支持人员]  [提交反馈]        │
└─────────────────────────────────────────────────────┘
```

**与系统级失败的视觉区分**：

- 系统级失败：图标使用工具/齿轮类图标，语调强调"技术问题，我们正在处理"
- 业务级失败：图标使用文档/感叹号类图标，语调强调"文件需要调整，请您处理"
- 两类失败均使用同一底色区（建议浅红/浅橙背景），但标题措辞有明确区分

#### 业务级失败后的数据处理

- ReviewSession.state 更新为 `aborted`
- 已提取的部分 ExtractedField 记录保留，但标记为 `incomplete`
- 合同文件保留 48 小时后按数据保留策略清理
- 用户重新上传新文件时，创建新的 ReviewSession，与原合同实体关联

---

### 5.4 用户引导操作规范

以下三类操作在所有失败场景中均需提供：

#### 重试

| 要素 | 规范 |
|------|------|
| 按钮文案 | 系统级失败："重试此步骤"；业务级失败："重新上传合同文件" |
| 可见条件 | 系统级失败始终显示；业务级失败中仅当原因为"扫描件质量"时显示，文件损坏类不显示 |
| 操作反馈 | 点击后按钮变为 loading 状态，同时更新步骤条节点状态为"进行中" |

#### 联系支持

| 要素 | 规范 |
|------|------|
| 按钮文案 | "联系技术支持" |
| 触发行为 | 打开支持渠道（邮件/内部工单系统），预填充以下信息：错误参考码、ReviewSession ID、发生时间 |
| 预填内容 | 格式示例：`[合同审核系统] 处理失败反馈 - 错误码: ERR-20260311-A8F2 - 会话: session_xxx` |

#### 提交反馈

| 要素 | 规范 |
|------|------|
| 按钮文案 | "提交问题反馈" |
| 触发行为 | 弹出行内反馈表单（非跳转新页面） |
| 表单字段 | 问题描述（文本框，必填）、是否允许团队查看文件（单选，可选）、联系邮件（选填） |
| 提交后反馈 | 显示"感谢您的反馈，已提交问题报告（编号: xxx）" |

## 六、状态流转定义

本章输出供架构设计和数据模型使用的状态枚举与转换条件，是本文档对后续阶段的核心交付物。

### 6.1 ReviewSession.state 完整枚举（本文档覆盖范围）

| 状态值 | 中文含义 | 触发进入条件 | 允许退出到 |
|--------|----------|-------------|-----------|
| `parsing` | 文件解析中 | 文件上传完成（Teammate 1 范围） | `scanning`、`aborted` |
| `scanning` | AI 风险扫描中 | 用户确认字段核对后触发扫描（或自动触发） | `hitl_pending`、`completed`（低风险直通）、`aborted` |
| `hitl_pending` | 等待人工介入 | LangGraph routing 节点完成路由判断 | `completed`、`aborted` |
| `completed` | 审核流程完成 | 高风险全部处理完 / 低风险自动通过后进入 | `report_ready`（子状态） |
| `aborted` | 流程已中止 | 任意阶段发生不可恢复的失败 | 无（终止状态）；用户可创建新 ReviewSession |

### 6.2 scanning → hitl_pending 的完整状态流转

#### 正常流转路径

```
scanning
  │
  ├─ [扫描进行中，LangGraph risk_scanning 节点执行]
  │
  ├─ [扫描完成，LangGraph routing 节点执行]
  │    │
  │    ├─ 无高风险，无中风险，低风险数量 ≤ 阈值
  │    │    └─→ completed（跳过 hitl_pending）
  │    │
  │    ├─ 无高风险，存在中风险
  │    │    └─→ hitl_pending（subtype: batch_review）
  │    │
  │    └─ 存在高风险（≥1 条）
  │         └─→ hitl_pending（subtype: interrupt）
  │
  └─ [扫描失败：LLM 超时 / 系统错误]
       └─→ aborted（system_failure）
```

#### hitl_pending 细分状态说明

`hitl_pending` 状态内部通过 `subtype` 字段区分，不作为顶层枚举扩展：

| subtype 值 | 触发条件 | 界面模式 | 允许的操作 |
|------------|----------|----------|-----------|
| `batch_review` | 中风险条款存在，无高风险 | 复核列表视图 | 允许批量确认 |
| `interrupt` | 高风险条款存在 | 流程暂停视图 | 禁止批量操作，必须逐条处理 |

### 6.3 状态流转的触发主体

| 状态转换 | 触发主体 | 触发方式 |
|----------|----------|----------|
| `parsing` → `scanning` | 用户（点击"开始扫描"）或系统（自动触发） | API 调用触发 LangGraph 节点推进 |
| `scanning` → `hitl_pending` | 系统（LangGraph routing 节点） | LangGraph 内部状态转换 + API 更新 ReviewSession |
| `scanning` → `completed` | 系统（LangGraph routing 节点，低风险路径） | 同上 |
| `hitl_pending` → `completed` | 用户（完成所有人工操作后）| Teammate 3 操作范围 |
| 任意状态 → `aborted` | 系统（不可恢复异常）或用户（主动放弃） | API 更新 ReviewSession.state = aborted |

### 6.4 操作追溯事件类型枚举

以下事件类型需在操作日志中记录，供审计使用：

| 事件类型 | 发生时机 |
|----------|----------|
| `field_verified` | 用户确认低置信度字段 |
| `field_modified` | 用户修改 AI 提取的字段值 |
| `field_verify_skipped` | 用户跳过低置信度字段核对 |
| `scan_triggered` | 风险扫描触发 |
| `scan_completed` | 风险扫描完成，路由结果已确定 |
| `route_auto_passed` | 低风险自动通过 |
| `route_batch_review` | 进入中风险批量复核队列 |
| `route_interrupted` | 高风险触发 LangGraph interrupt |
| `system_failure` | 系统级失败 |
| `business_failure` | 业务级失败 |
| `retry_triggered` | 用户触发重试 |
| `session_aborted` | 审核会话中止 |
