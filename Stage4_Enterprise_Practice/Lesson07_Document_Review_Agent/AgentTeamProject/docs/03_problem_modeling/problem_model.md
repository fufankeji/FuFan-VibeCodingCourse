## 一、核心业务问题定义

### 1.1 问题陈述

企业法务/采购人员在处理合同审核时，面临以下核心矛盾：

> **审核工作量与合规质量之间的结构性张力**：人工审核单份合同耗时 2-4 小时，在合同量增加时，审核人员只能压缩每份合同的精审时间，导致风险条款漏检率上升；而若维持质量则形成人力瓶颈，审核周期延误业务进展。

这一矛盾不能通过单纯的"自动化替代"来解决，原因在于：
- 合同条款的法律判断含有模糊性，需要人工介入确认
- 跨条款组合风险（如格式塔条款、条件触发义务）超出当前 LLM 的可靠推理能力
- 受监管行业的合规审计要求每一步操作都可追溯

**系统要解决的核心问题是**：在保持人工对关键决策的最终控制权的前提下，通过 AI 完成结构化信息提取与风险初筛，将人工精力聚焦在真正需要判断的高风险条款上，同时确保全流程操作留下可审计的状态记录。

### 1.2 问题拆解

| 子问题 | 当前痛点 | 系统预期解 |
|--------|----------|-----------|
| **信息提取效率低** | 审核人员手动从合同中提取主体、金额、日期等结构化信息，机械重复 | Agent 自动提取，结构化输出 |
| **风险条款定位费时** | 需逐行阅读全文才能找到潜在风险条款 | AI 初筛后按风险等级排序呈现，双向关联原文定位 |
| **人工审核无法聚焦** | 所有条款平等呈现，审核人员无法快速区分高/低风险 | 风险三档分级路由，高风险强制中断等待人工 |
| **流程状态不透明** | 合同在多人手中流转时，无法实时了解审核进度和当前卡点 | LangGraph StateGraph 工作流状态可视化 |
| **结果无审计链路** | 审核结论缺乏操作记录，无法溯源"谁在何时基于何判断做出决定" | Checkpointer 持久化，每步操作记录 |
| **AI 结论无法信任** | 审核人员不清楚 AI 判断的依据，难以建立使用信心 | 置信度标识 + 规则来源标注（规则触发 vs AI 推理区分） |

## 二、目标用户

### 2.1 用户角色定义

本系统 MVP 阶段面向企业内部合同审核场景，核心用户为：

#### 主用户：法务合同审核人员

- **职责**：负责合同文本的风险识别与合规判断，给出是否签署的建议
- **使用频率**：每日处理多份合同，高频用户
- **核心诉求**：快速定位风险条款、减少重复性信息提取工作、保留对最终判断的控制权
- **关键痛点**：审核量大时精力无法集中在真正高风险的部分；AI 建议不透明时不敢依赖
- **典型行为**：上传合同 → 查看 AI 初筛结果 → 对高风险条款逐一确认（Approve/Edit/Reject）→ 导出报告

#### 次要用户：业务方/合同发起人

- **职责**：发起合同审核需求，关注审核进度和结论
- **使用频率**：按需，低频
- **核心诉求**：知道合同在哪个审核节点、预计何时完成
- **关键行为**：提交合同 → 跟踪审核状态 → 接收审核报告结论

#### 系统管理员

- **职责**：维护规则库、配置风险阈值、管理用户权限
- **使用频率**：低频，配置类操作
- **核心诉求**：能够配置和更新企业内部合同审核标准（Playbook/规则库）

### 2.2 用户范围约束（MVP 阶段）

以下用户场景在 MVP 阶段**不纳入**：

- 律师事务所对外服务场景（合规要求和使用模式差异大）
- 跨境合同审核（多法域叠加风险超出当前 Agent 能力）
- 强监管行业的重大合同（如金融衍生品、工程总承包，属于已识别的禁区）

## 三、MVP 业务场景

MVP 阶段对应 01 阶段建议的 **Phase 1**：合同信息结构化提取 + 格式合规检查，同时包含风险初筛的 HITL 核心链路。

### 场景 1：合同信息结构化提取

**触发条件**：审核人员上传一份 PDF 或 DOCX 格式的合同文件

**核心流程**：
1. 系统接收文件，调用 OCR（外部成熟方案）完成文本解析
2. Agent 提取结构化字段：合同主体（甲/乙方）、合同金额、生效日期、终止条件、签署方信息
3. 提取结果以结构化表单形式呈现，标注字段提取的置信度
4. 低置信度字段（如 OCR 错误导致）以醒目标识提示人工核对

**业务价值**：消除手动录入工作，将结构化信息提取时间从 20-40 分钟压缩至 1-2 分钟

**成功标准**：主要字段提取准确率可在真实合同库中建立基线（不依赖通用 benchmark）

---

### 场景 2：风险条款 AI 初筛 + 分级路由

**触发条件**：结构化提取完成后，系统自动进入风险扫描阶段

**核心流程**：
1. Agent 对合同全文进行风险条款扫描，覆盖：单边条款、兜底条款滥用、违约金不对等、弹性表述
2. 每条风险判断附带：风险等级（高/中/低）、置信度、判断来源（规则触发 or AI 推理）、原文定位锚点
3. 分级路由执行：
   - **低风险**：自动标记通过，进入批量复核队列
   - **中风险**：进入人工复核列表，允许批量确认
   - **高风险**：强制中断（LangGraph `interrupt`），审核人员必须逐条处理后流程才能继续

**约束**：
- AI 判断必须附带不确定性标识，禁止以"审核完成"等绝对化表述呈现结果
- 高风险条款确认必须展开原文 + 填写接受原因（防止 Automation Bias）

**业务价值**：将审核人员的精力从"通读全文"聚焦到"处理高风险条款"，预期精审时间从 2-4 小时压缩至 ≤30 分钟

---

### 场景 3：HITL 人工决策与流程恢复

**触发条件**：审核流程在高风险节点触发 `interrupt`，等待人工处理

**核心流程**：
1. 系统呈现"双栏对照视图"：左侧为风险条款摘要和 AI 判断，右侧为合同原文（锚定到对应段落）
2. 审核人员对每条高风险条款执行三选项操作：
   - **Approve**：确认 AI 判断，标注接受理由
   - **Edit**：修改 AI 的风险描述或等级，记录修改内容
   - **Reject**：标记 AI 判断有误，标注理由（作为规则库反馈数据）
3. 所有操作通过 Checkpointer 持久化，支持异步跨天恢复（审核人员中断后可从中断点继续）
4. 全部高风险条款处理完成后，流程自动 `resume` 进入报告生成阶段

**约束**：高风险条款不允许批量一键通过，必须逐条展开原文确认

**业务价值**：
- 异步恢复机制解决了当前竞品中几乎无对标的"跨天审核中断"问题
- 操作记录满足受监管行业的审计可追溯要求

---

### 场景 4：审核报告生成与导出

**触发条件**：所有条款审核（含人工决策）完成后

**核心流程**：
1. 系统自动生成审核报告，包含：
   - **执行摘要**：合同基本信息、整体风险评估、审核结论
   - **详细分析**：逐条风险清单（含 AI 判断 + 人工最终决策 + 处理记录）
   - **覆盖范围声明**：明确标注"本次审核覆盖的条款类型"和"未覆盖的条款类型"
2. 报告导出格式：PDF（归档用）、JSON（系统集成用）
3. 报告中所有 AI 分析结论附加免责声明："AI 辅助分析，最终判断由人工审核人员负责"

**业务价值**：结构化报告替代人工撰写审核纪要，同时提供可机读的 JSON 格式支持下游系统集成

## 四、领域模型（核心实体与关系）

以下实体是后续数据模型设计的基础：

```
Contract（合同）
  ├── id, title, file_path, file_type, status
  ├── uploaded_by (-> User), uploaded_at
  └── 1:1 → ReviewSession

ReviewSession（审核会话）
  ├── id, contract_id, state (枚举: parsing | scanning | hitl_pending | completed | aborted)
  ├── langgraph_thread_id          # 对应 LangGraph checkpointer 的线程标识
  ├── created_at, completed_at
  └── 1:N → ReviewItem

ReviewItem（审核条款）
  ├── id, session_id
  ├── clause_text                  # 原文段落
  ├── clause_location              # 原文定位锚点（页码 + 段落偏移）
  ├── risk_level (low | medium | high)
  ├── confidence_score             # 置信度 0-100
  ├── source (rule_engine | ai_inference)  # 判断来源
  ├── ai_finding                   # AI 的风险描述
  ├── human_decision (approve | edit | reject | pending)
  ├── human_note                   # 人工填写的接受原因或修改内容
  ├── decided_by (-> User), decided_at
  └── is_false_positive            # Reject 时标记为误报

ExtractedField（结构化字段）
  ├── id, session_id
  ├── field_name (party_a | party_b | amount | effective_date | termination_condition | ...)
  ├── field_value
  ├── confidence_score
  └── needs_human_verification     # 低置信度标记

ReviewReport（审核报告）
  ├── id, session_id
  ├── summary_json                 # 执行摘要
  ├── detail_json                  # 详细条款清单
  ├── coverage_statement           # 覆盖范围声明
  ├── pdf_path, json_path
  └── generated_at

User（用户）
  ├── id, name, role (reviewer | submitter | admin)
  └── email
```

**核心状态机（ReviewSession.state）**：

```
[上传] → parsing → [解析完成] → scanning → [扫描完成] → hitl_pending
  ↓ 高风险条款全部处理完成
completed → [报告生成] → report_ready
  ↓ 任意阶段可触发
aborted
```

## 五、核心约束（设计红线）

以下约束来自 01、02 两阶段的共识，必须在架构设计和原型设计中贯彻：

| 约束类型 | 约束内容 | 来源 |
|----------|----------|------|
| **人工决策不可绕过** | 高风险条款必须逐条人工处理，不提供批量通过入口 | 01 阶段 HITL 核心共识 |
| **AI 结论不可绝对化** | 所有 AI 判断必须附带置信度，报告中必须包含免责声明 | 02 阶段设计禁区 |
| **覆盖范围必须声明** | 报告中必须标注"已覆盖"和"未覆盖"的条款类型 | 02 阶段 S9 |
| **判断来源必须可区分** | 界面上明确区分"规则触发"与"AI 推理"两种来源 | 02 阶段共识 2 |
| **操作全程可追溯** | 每步操作通过 Checkpointer 持久化，满足审计要求 | 01 阶段合规目标 |
| **不自研 OCR** | 接入外部成熟 OCR 方案，禁止自研 | 02 阶段设计禁区 |
| **不做完整 CLM 平台** | 聚焦审核引擎核心链路，不扩展至电子签章、合同模板等 CLM 功能 | 02 阶段设计禁区 |
| **数据安全** | 合同文件不得明文传输至不受控的外部 API，部署方案需考虑数据主权 | 01 阶段硬约束 |
