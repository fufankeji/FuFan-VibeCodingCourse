## 一、场景目标（从风险识别角度重新定义）

### 1.1 场景的实际价值主张

"合同/法务类文档 Agent 审核"的核心价值主张是：用 AI 替代或辅助人工，在更短时间内识别合同中的风险条款，降低法律风险，提升企业合规效率。

这个主张在表述层面是成立的。但在落地层面，需要拆解以下两个关键前提是否为真：

- **前提 A**：Agent 识别风险的准确率足够高，不会制造新的风险（误报/漏报风险）
- **前提 B**：引入 Agent 后的整体流程，比纯人工流程在风险控制上更优，而不只是更快

**反方核心论点**：目前的技术水平下，前提 A 存在显著的不确定性，而业界对前提 B 的验证几乎是空白的。在无法证伪这两个前提之前，盲目上线 Agent 审核系统，其带来的风险可能远超其节省的成本。

### 1.2 风险识别的边界：Agent 能做什么，不能做什么

| 能力维度 | Agent 的实际边界 | 常见的过度承诺 |
|---------|----------------|-------------|
| 关键词/模式匹配 | 可以高置信度完成 | 被包装为"全面风险识别" |
| 标准格式合同审核 | 表现尚可 | 被推广到非标准、定制化合同 |
| 单条款语义理解 | 基本可用 | 被用于跨条款组合风险判断 |
| 跨法域差异识别 | 极度不可靠 | 被用于国际合同或多法域合同 |
| 隐含义务与默示条款 | 基本无能力 | 几乎从不被提及 |
| 法律责任归属判断 | 超出模型能力范围 | 以"风险评分"形式模糊呈现 |

### 1.3 反方审查的核心关切

本报告不反对使用 Agent 技术，但坚持认为：**任何合同 Agent 审核系统在上线前，必须先回答清楚它在什么情况下会出错，而不只是展示它能在什么情况下做对。**

## 二、审核对象中的高风险环节

### 2.1 条款层级：最容易出错的合同类型

以下条款类型是 Agent 误报和漏报的高发区，按风险等级排列：

**极高风险（漏报代价极大）**

- **格式塔条款（Gestalt Clauses）**：单独看每条都合法，但组合起来构成隐性不平等。例如：免责条款 + 仲裁地选择 + 适用法律选择三者叠加，可能让一方完全丧失实质性救济权利。当前 LLM 的上下文窗口虽然在扩大，但跨条款的逻辑推理能力远弱于单条款理解能力。
- **条件触发义务**：合同中大量"若……则……"结构，在特定业务场景下才会激活的条款，Agent 倾向于孤立评估其表面语义，而非结合业务逻辑判断其触发可能性。
- **兜底条款与弹性义务**："合理努力"、"商业上可行"等弹性表述，法律争议极大，Agent 几乎无法给出有意义的风险判断，但可能以高置信度输出一个无意义的结论。

**高风险（误报率高，导致审核疲劳）**

- **标准行业条款**：被 Agent 过度标注为风险的可能性很高。例如，在 IT 服务合同中，"服务商不对间接损失承担责任"是行业惯例，但可能被 Agent 持续标红，消耗审核人员注意力。
- **交叉引用条款**：引用另一合同、附件或外部标准的条款，Agent 通常无法访问被引用文件，会产生基于不完整信息的错误判断。

### 2.2 文档层级：输入质量对结果的毁灭性影响

Agent 的输出质量对输入格式高度敏感，但这一点在产品演示中几乎从不被提及：

- **扫描件与 OCR 错误**：大量实际业务中的合同是扫描 PDF。OCR 识别错误（如"乙方"被识别为"已方"、金额数字识别错误）会直接导致 Agent 在错误的文本上进行分析，产生系统性的静默错误——既无报错，也无警告。
- **多语言混合文档**：中英双语合同中，Agent 可能在两种语言版本之间产生理解偏差，而两个版本在法律上的优先顺序本身就是一个风险点。
- **非标准排版与嵌套结构**：复杂的表格、注脚、附件引用、条款编号不规范等，会导致 Agent 对条款归属和层级关系产生误判。

### 2.3 流程层级：审核链路中的信息损失节点

将 Agent 插入审核流程后，会在以下节点产生信息损失：

```
原始合同文本
    → [文本提取/OCR] ← 信息损失节点 1：格式转换错误
    → [分块/上下文截断] ← 信息损失节点 2：长文档超出上下文窗口
    → [Agent 分析] ← 信息损失节点 3：模型幻觉与过度自信
    → [结果摘要输出] ← 信息损失节点 4：风险被平铺化、等级化，丧失细节
    → [人工审核] ← 信息损失节点 5：审核人依赖摘要而非原文
    → [决策]
```

每个节点的信息损失是累积的，且大多数是不可逆的。在最终决策时，决策者实际上是在基于经过五次过滤的信息做判断，而不是基于合同原文。

## 三、为什么需要 LangChain 做人机交互流程（从规避风险角度）

### 3.1 纯自动化审核的风险敞口

在没有 Human-in-the-loop 机制的情况下，一个全自动的合同审核 Agent 存在以下无法接受的风险：

- Agent 输出的风险判断直接进入决策流程，没有任何人类校验层
- 当 Agent 发生系统性偏差（如对某类条款持续漏报）时，无法被及时发现
- 所有错误的法律和商业后果由使用方承担，而 Agent 系统本身无法承担法律责任
- 批量处理大量合同时，单次错误会被快速复制，损失呈指数级放大

**结论**：合同审核场景的风险代价不对称性极高（漏报一个关键条款可能导致数百万甚至更大的损失），因此纯自动化审核在当前技术条件下是不负责任的。

### 3.2 LangGraph Human-in-the-loop 机制的结构性作用

根据 LangChain 官方文档，LangGraph 的 Human-in-the-loop (HITL) 机制提供了以下关键能力：

**中断与等待机制（Interrupt）**

LangGraph 的 `interrupt()` 函数允许在图执行的任意节点暂停，并将当前图状态通过 Persistence Layer（持久化层）保存。这意味着：

- 审核流程可以在 Agent 标记出高风险条款后，暂停并等待法务人员确认，而不是自动执行下一步动作
- 暂停期间图状态被持久化（基于 checkpointer），支持无限期等待，适配法务审核的实际工作节奏
- 人工审核后可以选择 `approve`（批准）、`edit`（修改 Agent 的判断）、或 `reject`（拒绝 Agent 的结论）

```
Agent 分析完成
    → interrupt() 触发
    → 图状态持久化到 checkpointer（生产环境推荐 PostgresStore）
    → 人工审核界面展示风险条款
    → 人工决策：approve / edit / reject
    → Command(resume=...) 恢复执行
    → 后续流程继续
```

**可配置的中断策略**

通过 `HumanInTheLoopMiddleware` 的 `interrupt_on` 参数，可以精细化配置哪些工具调用必须经过人工审批：

```python
# 示例：仅对高风险操作触发人工审核
HumanInTheLoopMiddleware(
    interrupt_on={
        "flag_critical_clause": True,    # 标记关键条款时必须人工审核
        "generate_summary": False,        # 生成摘要时自动通过
        "extract_parties": False,         # 提取合同方信息时自动通过
    }
)
```

这种分级策略理论上可以避免对每个 Agent 动作都要求人工审批，从而缓解审核瓶颈。

### 3.3 HITL 机制的必要性论证

从风险规避角度，HITL 机制在合同审核场景中是结构性必要条件，而非可选的锦上添花功能。理由如下：

1. **法律责任隔离**：有了人工审核节点，最终的审核责任由人类承担，Agent 降级为辅助工具而非决策主体，在合规和法律层面具有本质区别
2. **误报/漏报的动态发现**：人工审核的反馈可以形成数据闭环，持续发现 Agent 的系统性偏差，为模型迭代提供训练数据
3. **知识领域补偿**：法律专业知识的深度和时效性超出任何当前 LLM 的能力边界，人工审核提供了专业知识的兜底层

### 3.4 反方质疑：LangChain 是否真正解决了根本问题

**LangGraph HITL 机制解决的是流程工程问题，而非认知质量问题。**

具体而言，LangGraph 的 interrupt/resume 机制保证了"流程上有人工介入的机会"，但它无法保证：

- 人工审核人员真正理解了 Agent 给出的风险标注的含义
- 人工审核人员不会因为 Agent 输出的高置信度表述而倾向于直接批准（自动化偏见 / Automation Bias）
- 人工审核的质量不会随着工作量增加而下降（审核疲劳问题，见第四章）

换言之，LangGraph 提供了一个装置良好的安全阀，但这个安全阀的实际效果，取决于阀门后面的人是否有能力和意愿做出独立判断。这是一个组织管理问题，不是一个技术问题，LangChain 无法解决它。

## 四、技术难点清单与应对建议

### 4.1 难点矩阵

| 技术难点 | 严重程度 | 当前可解决度 | 备注 |
|---------|---------|------------|------|
| 误报率控制 | 高 | 中 | 可通过领域微调和阈值调整缓解，但无法消除 |
| 漏报率控制 | 极高 | 低 | 漏报的代价不对称，当前技术无充分保障 |
| 跨条款组合风险识别 | 极高 | 极低 | 需要完整的法律推理能力，超出现有 LLM 边界 |
| 长文档上下文截断 | 高 | 中 | 扩大上下文窗口可缓解，但成本和延迟显著增加 |
| 扫描件 OCR 静默错误 | 高 | 低 | OCR 错误无法被 Agent 自动检测和修正 |
| 人工审核自动化偏见 | 高 | 低 | 属于认知科学问题，技术手段效果有限 |
| 审核疲劳导致质量下滑 | 高 | 中 | 可通过优化中断策略减少无效打扰 |
| 跨法域合规差异 | 极高 | 极低 | 需要持续更新的法律知识库，维护成本极高 |
| LLM 幻觉与过度自信 | 高 | 中 | 可通过输出格式约束和不确定性表达缓解 |
| 审核数据隐私与主权 | 高 | 中 | 私有化部署可解决，但增加基础设施复杂度 |
| HITL 持久化存储稳定性 | 中 | 高 | LangGraph checkpointer 生产环境需 PostgresStore，开发文档已明确 |

### 4.2 误报率：被低估的业务损耗

误报（将合法条款标记为风险）的直接危害容易被忽视，因为它不会造成直接的法律损失。但误报带来的间接损耗是系统性的：

**误报率的累积效应模型**

假设一份合同有 100 个条款，Agent 的整体误报率为 10%（行业内这个数字通常更高），则：

- 每份合同平均产生 10 个需要人工排查的误报
- 如果每天处理 50 份合同，则每天产生 500 个需要确认"这不是真正风险"的审核任务
- 审核人员在重复确认误报的过程中，注意力被消耗，对真正风险条款的识别能力下降
- 这就是"误报诱导的漏报"——误报率高会间接导致漏报率升高

**这是一个目前没有好的技术解决方案的问题。** 降低误报率的常见手段（提高判断阈值）会直接拉高漏报率，二者存在根本性的权衡。

### 4.3 漏报率：无法被系统自动发现的失效模式

漏报的核心危险在于其**不可见性**：

- 系统运行正常，没有报错，没有警告
- 审核人员看到 Agent 输出的清单，认为已经审核完毕
- 真正的风险条款被静默地放过
- 损失在合同执行阶段才显现，此时通常为时已晚

漏报的发现依赖于：一是事后的合同执行纠纷（成本极高），二是周期性的人工全量抽检（成本也不低）。两种方式都无法提供实时保障。

**应对建议**：
- 对高价值合同（金额超过阈值、涉及核心业务条款）强制要求人工全量审核，Agent 仅作为辅助标注工具
- 建立漏报追踪机制：当合同在执行阶段产生纠纷时，回溯 Agent 的审核记录，统计漏报类型，形成持续改进闭环
- 在系统界面上明确展示 Agent 的覆盖率声明（"本次审核覆盖了哪些条款类型"），而非模糊的"审核完成"

### 4.4 人工审核瓶颈的本质与缓解策略

**问题本质**

引入 Agent HITL 流程后，人工审核的工作性质发生了变化：从"主动阅读合同发现问题"变成"被动响应 Agent 的打扰请求"。这个变化表面上减轻了工作量，但实际上：

- 审核人员失去了自主阅读的节奏感，难以建立对整份合同的整体理解
- 被打扰的频率越高，每次打扰的认知深度越低
- 当 Agent 的误报率较高时，审核人员会逐渐养成"习惯性批准"的行为模式

**这与 LangGraph interrupt 机制设计的初衷恰恰相反**：interrupt 的设计假设是人工审核是有价值的、高质量的，但实际使用中，高频的低质打扰会让 interrupt 机制退化为一个橡皮图章。

**缓解策略**

- **分级中断策略**：仅对 Agent 判断置信度低于阈值的条款触发中断，高置信度结果直接记录，定期批量复核
- **批量审核模式**：将同一类型的风险条款聚合后一次性呈现，而不是逐条打断
- **上下文完整性保障**：中断时向审核人员展示的不应只是被标注的条款，而应包含足够的上下文（前后条款、合同背景）
- **审核工作量监控**：建立每日人工审核量的上限警戒线，超出上限时自动降级为"仅接受高置信度风险标注"模式

### 4.5 法律合规风险的兜底机制

**责任归属问题没有技术解**

目前没有任何国家的法律框架对"AI 辅助合同审核导致损失"的责任归属有明确规定。在这一法律真空中，最可能的结果是：责任由使用 Agent 的企业或法务人员承担，而与 Agent 供应商无关（通过服务条款的免责声明）。

**最低限度的合规保障要求**

在法律框架明确之前，部署合同审核 Agent 的企业至少应当：

1. 在审核记录中完整保存 Agent 的每次输出，以及人工审核人员的每次决策，形成可追溯的审计日志
2. 明确内部政策：Agent 的审核结论不构成法律意见，不能替代律师的专业判断
3. 对外部合同（与第三方签署的合同）和内部文件（内部制度文件）采用不同的审核标准

### 4.6 数据安全：被忽视的隐形风险

合同文档包含企业最敏感的商业信息：价格、条款、战略合作方、知识产权边界。当这些文档被发送到外部 LLM API（如 OpenAI、Anthropic）时：

- 数据在传输过程中存在泄露风险（即使有 TLS 加密，API 服务商的日志策略不透明）
- 部分 API 服务商的用户协议允许将用户数据用于模型训练，尽管企业合同数据通常被排除在外，但需要明确确认
- 数据主权问题：合同数据可能被存储在境外服务器，与某些行业的监管要求冲突

**应对建议**：优先考虑本地部署或私有化部署方案，即使牺牲部分模型能力。对于涉及国家秘密、商业机密或受监管行业（金融、医疗）的合同，外部 API 调用方案应被直接排除。

### 4.7 尚无成熟解决方案的遗留问题

以下问题在本文撰写时（2026-03-11）尚无业界公认的成熟解决方案，需要诚实地向决策层说明：

1. **法律语言的组合语义推理**：当前 LLM 无法可靠地推断多条款组合后的法律效果
2. **实时法律知识更新**：法律法规持续变化，模型的训练截止日期是固定的，无法保证对最新法律变化的覆盖
3. **Automation Bias 的技术性干预**：没有证据表明任何 UI 设计或工作流设计能有效对抗审核人员对 AI 结论的过度依赖
4. **跨司法管辖区的差异化处理**：国际合同中涉及的法域差异（大陆法系 vs. 普通法系、不同国家的强制性规定）超出通用模型的可靠处理范围

## 五、结论：建议的风险接受边界

### 5.1 哪些场景可以部署 Agent（风险可接受）

以下场景中，Agent 辅助审核的收益大于风险，可以在配备完整 HITL 机制的前提下部署：

| 适用场景 | 理由 |
|---------|------|
| 标准化合同的格式合规检查 | 规则明确，误报率低，漏报代价有限 |
| 合同信息提取（合同方、金额、期限） | 属于结构化信息提取，准确率较高 |
| 内部低价值合同的初筛 | 不进入最终法律决策，人工兜底成本低 |
| 大量历史合同的分类与归档 | 即使有误差，不影响实际业务决策 |
| 为律师提供条款检索和对比参考 | 律师保留最终判断权，Agent 仅作检索工具 |

### 5.2 哪些场景必须保留全量人工审核

以下场景中，当前技术水平下不应让 Agent 介入实质性审核决策：

| 禁用场景 | 理由 |
|---------|------|
| 重大经济合同（金额 > 内部设定阈值） | 漏报的经济损失不可接受 |
| 跨境合同与国际商事合同 | 法域差异超出模型可靠处理范围 |
| 涉及知识产权许可与转让的合同 | 条款的组合效果复杂，漏报代价极高 |
| 受强监管行业的合同（金融、医疗、能源） | 合规要求明确，违规代价包括刑事责任 |
| 任何将成为诉讼证据的合同 | 法律效力要求必须有人类专业人士背书 |

### 5.3 给 Lead 的决策建议

**建议一：先做能力边界测试，再谈上线**

在正式上线前，应当用真实的历史合同（已知最终结果的）对 Agent 进行盲测，统计实际误报率和漏报率，而不是依赖供应商的演示数据或通用 benchmark 数据。合同法律领域的性能与通用 NLP 任务的性能不具有可比性。

**建议二：HITL 不是技术配置，而是组织承诺**

LangGraph 的 interrupt/resume 机制可以在技术上实现人工介入节点，但这只是必要条件而非充分条件。真正有效的 HITL 要求：明确的人工审核 SLA、审核人员的专业培训（包括对抗 Automation Bias 的培训）、以及对审核质量的独立监控机制。如果这些组织配套无法到位，HITL 机制将形同虚设。

**建议三：从最低风险的场景开始，建立真实的性能基线**

不要试图一步到位地替代所有人工审核。建议从"格式合规检查"和"合同信息结构化提取"这两个技术风险最低的场景切入，在真实业务环境中积累 6-12 个月的性能数据，建立针对本企业合同库的真实误报率和漏报率基线，再以此为依据决定是否扩展到更高风险的场景。

**建议四：明确不上线 Agent 的退出标准**

在立项时就应当定义清楚：如果上线后发现哪些指标不达标，项目将被暂停或下线。缺乏退出标准的 AI 项目，会因为沉没成本的压力而持续运行，即使已经产生了明显的风险。
