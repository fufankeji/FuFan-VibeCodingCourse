## 一、场景目标

### 1.1 业务背景与驱动力

企业在日常经营中持续产生大量合同文档。以一家中型制造企业为例，每年需审核的合同数量可达数百乃至数千份，涵盖采购框架协议、销售合同、工程承包合同、劳动合同、保密协议（NDA）、知识产权授权协议等多种类型。传统的合同审核工作高度依赖法务人员的人工阅读与经验判断，在以下几个维度存在明显瓶颈：

- **效率瓶颈**：一名有经验的法务律师审核一份中等复杂度的商业合同平均耗时 2-4 小时，高并发场景下极易形成积压。
- **一致性缺失**：不同审核人员对同一合同的风险判断存在主观差异，缺乏标准化的审核结论。
- **知识覆盖局限**：法务人员难以同时熟悉所有业务领域（IT 采购、金融衍生品、跨境贸易等）的专业条款规范。
- **版本追踪困难**：多轮谈判后的版本变更记录分散，差异对比耗时且易遗漏。

引入 Agent 技术的驱动力来自三个层面：**降本增效**（减少重复性阅读工作）、**提升合规**（系统化识别风险条款）、**知识沉淀**（将审核经验转化为可复用的规则库与案例库）。

### 1.2 核心目标定义

| 目标维度 | 当前状态 | 引入 Agent 后的预期 |
|----------|----------|---------------------|
| 审核周期 | 单份合同 2-4 小时 | 初筛报告 < 5 分钟，人工复核 < 30 分钟 |
| 风险覆盖率 | 依赖个人经验，约 70-80% | 规则库 + LLM 联合检测，目标 > 95% |
| 审核标准一致性 | 因人而异 | 基于统一规则库，结论可追溯、可复现 |
| 合规知识覆盖 | 受限于团队专业背景 | 可接入行业法规、判例数据库进行实时比对 |

### 1.3 适用的组织类型与规模

- **大型企业法务部门**：高并发合同量，需要批量初筛 + 法务人员精审的分层审核模式。
- **中小企业（无专职法务）**：借助 Agent 辅助非法律专业人员完成基础合规检查。
- **律师事务所/法律科技公司**：作为增值服务工具，提升客户服务效率。
- **金融、医疗、政府采购等强监管行业**：合规要求高，需要系统化、可留痕的审核流程。

## 二、审核的对象

### 2.1 合同文档类型分类

合同审核场景中，文档类型决定了审核策略与风险关注点的差异。以下是主要分类：

| 类别 | 典型文档 | 核心审核关注点 |
|------|----------|----------------|
| 商业采购类 | 采购框架协议、供应商合同 | 价格条款、交货期、质保责任、违约金 |
| 销售服务类 | 销售合同、服务协议、SLA | 服务范围、验收标准、知识产权归属 |
| 劳动用工类 | 劳动合同、竞业限制协议 | 试用期条款、竞业范围、赔偿标准合规性 |
| 保密类 | NDA、技术保密协议 | 保密范围定义、保密期限、信息披露例外 |
| 知识产权类 | 专利授权、软件许可协议 | 授权范围、使用限制、版权归属、许可费 |
| 金融类 | 贷款协议、担保合同、融资租赁 | 利率条款、违约事件定义、抵押条款 |
| 工程建设类 | 施工合同、EPC 总承包合同 | 工期条款、变更索赔、争议解决机制 |
| 数据合规类 | 数据处理协议（DPA）、隐私协议 | GDPR/个人信息保护法合规性、数据主权 |

### 2.2 审核要素清单

无论合同类型如何，以下要素构成审核的核心维度：

**结构性信息提取（必须项）**

- 合同主体：甲方、乙方、第三方担保人的全称、主体资格
- 合同标的：服务/商品描述、数量、规格
- 金额与支付：总价、分期付款安排、付款条件、货币种类
- 时间节点：签约日期、履行期限、交付截止日、有效期
- 管辖与适用法律：争议解决方式（仲裁/诉讼）、适用法律、管辖地

**风险条款识别（高优先级）**

- **单边不公平条款**：仅对一方施加严苛义务而另一方无对等约束
- **兜底条款滥用**："其他情形由甲方单方决定"类表述
- **知识产权归属模糊**：工作成果所有权未明确约定
- **无限连带责任**：担保范围超出通常商业惯例
- **自动续约陷阱**：续约通知期过短（如 3 天内须书面通知）
- **不可抗力范围过窄/过宽**：影响一方合法抗辩权
- **违约金显失公平**：远超实际损失或违反法定上限

**合规性校验**

- 是否符合当地劳动法强制条款
- 数据条款是否满足《个人信息保护法》/GDPR 要求
- 金融条款是否符合监管利率上限
- 格式条款是否依法进行了提示说明（消费合同场景）

### 2.3 典型合同结构说明

一份标准商业合同通常由以下结构化部分组成，Agent 需能识别并映射每个部分：

```
合同文档结构
├── 首部（当事人信息、背景陈述）
├── 定义条款（关键术语定义）
├── 核心义务条款
│   ├── 甲方义务
│   └── 乙方义务
├── 价款与支付条款
├── 知识产权条款
├── 保密条款
├── 违约责任条款
├── 合同解除条款
├── 不可抗力条款
├── 争议解决条款
└── 附件（技术规范、报价单等）
```

Agent 的文档解析目标是将上述非结构化文本转化为结构化的审核对象，便于后续规则匹配与风险评级。

## 三、为什么需要 LangChain 做人机交互的流程

### 3.1 纯自动化的根本局限：法律判断的不可完全自动化性

合同审核不同于代码 Lint 检查或数据格式校验——它本质上是一种**需要结合上下文、商业意图与法律判断的专业推理活动**。以下几类问题无法通过纯自动化规则引擎可靠解决：

**模糊性判断**：同一条违约金条款，在不同行业背景、不同合同金额量级下，"是否显失公平"的结论截然不同。这类判断需要专家在 Agent 给出初步分析后进行最终裁定。

**新型条款识别**：法律实践不断演进，合同中出现的新型数字资产条款、AI 生成内容归属条款、碳排放权转让条款等，可能超出预置规则库的覆盖范围。LLM 能识别异常，但对其法律效力的判断需要人工参与。

**商业意图确认**：某些条款在文本层面存在风险信号，但可能是双方协商的合理安排。例如，宽泛的免责条款可能已在价格谈判中得到补偿。Agent 无法独立判断这种商业层面的权衡，需要业务方确认。

**可审计性要求**：在金融、医疗、政府采购等受监管领域，审核结论必须有人工签字确认的留痕记录。即便 Agent 的判断完全正确，合规程序也要求人工节点的存在。

**结论是**：合同审核是一个天然需要"人机协作"而非"全自动化"的场景，人工介入不是技术缺陷的补丁，而是业务流程的设计要求。

### 3.2 LangChain 在多步骤工作流中的角色

LangChain 提供了构建合同审核 Agent 所需的核心基础设施：

**文档加载与解析层**：LangChain 的 `Document Loader` 抽象提供了标准化的接口，支持 PDF、Word（DOCX）、纯文本等多种合同格式的加载。每个 `Document` 对象携带 `page_content`（文本内容）和 `metadata`（文件名、页码等元数据），为后续处理提供统一的数据格式。

**语义检索层（RAG）**：合同审核规则库（如公司内部审核标准、历史风险案例、行业监管要求）不适合全量塞入 LLM 上下文。LangChain 的 Embedding + Vector Store 抽象支持将规则库向量化存储，在审核时通过语义相似性检索最相关的规则片段，以 RAG 方式增强 LLM 的判断依据。

**结构化输出层**：LangChain 的 `Structured Output` 能力允许 Agent 以预定义的 Pydantic 模型返回审核结果（如风险条款列表、风险等级、建议措辞），而非自由文本，便于下游系统消费和展示。

### 3.3 LangGraph 状态机在审核流转中的核心价值

LangGraph 是构建合同审核工作流的关键框架。根据官方文档，LangGraph 的核心设计围绕三个原语：**State（共享状态）、Nodes（节点函数）、Edges（路由逻辑）**。

在合同审核场景中，这三个原语的映射关系如下：

| LangGraph 原语 | 合同审核场景映射 |
|----------------|-----------------|
| State | 审核任务的完整上下文：合同文本、已识别风险列表、当前审核阶段、人工批注记录、最终结论 |
| Nodes | 各审核步骤的执行函数：文档解析节点、风险扫描节点、规则匹配节点、人工审核暂停节点、报告生成节点 |
| Edges | 审核流程的路由逻辑：条件边（有重大风险则路由至人工节点，无风险则直接生成报告）|

**持久化执行（Durable Execution）的价值**：

合同审核不是一个即时完成的任务。法务人员可能在审核到一半时需要查阅外部资料，或者等待业务方确认商业背景，这个过程可能跨越数小时乃至数天。LangGraph 的持久化层（Checkpointer）在图执行的每个步骤后保存完整的 State 快照。即便系统重启或人工审核延迟一周后才响应，工作流也能从中断点精确恢复，而不会丢失任何已完成的分析结果。

官方文档明确指出：

> "By preserving completed work, durable execution enables a process to resume without reprocessing previous steps — even after a significant delay (e.g., a week later)."

这对合同审核场景尤为关键，因为审核周期本身就可能是跨天的异步流程。

### 3.4 人机协作节点设计：何时触发人工介入

LangGraph 通过 `interrupt` 机制实现工作流的暂停与恢复。根据官方文档，`HumanInTheLoopMiddleware` 支持三种决策类型：

- **approve（批准）**：接受 Agent 的分析结论，继续执行后续步骤
- **edit（修改）**：人工修改 Agent 的判断内容后继续
- **reject（驳回）**：否定 Agent 的结论，并可附加反馈说明，指导 Agent 重新分析

在合同审核场景中，应当触发人工介入的情形包括：

| 触发条件 | 风险等级 | 人工介入类型 |
|----------|----------|--------------|
| Agent 识别到"高风险"条款 | 高 | 强制人工确认（approve/reject/edit） |
| Agent 对某条款置信度低于阈值 | 中 | 建议人工复核（可跳过） |
| 合同金额超过预设阈值（如 100 万） | 高 | 强制法务总监级别审批 |
| 涉及跨境条款、外资监管 | 高 | 强制合规专员审核 |
| 首次出现的新型条款类型 | 中 | 人工判断并标注入库 |
| 合同主体信息存在异常（如疑似虚假主体） | 高 | 暂停并触发尽职调查流程 |

**关键设计原则**：中低风险条款允许 Agent 自主生成建议，人工仅对高风险节点介入，避免"全程审批"导致的效率损失。

## 四、具体的应用方案

### 4.1 整体架构：多 Agent 分工协作

合同审核系统采用多 Agent 分工架构，各 Agent 角色职责明确：

```
合同审核系统 Agent 架构

  [文档输入层]
       |
  ┌────▼────────────────────────────────────────┐
  │  解析 Agent                                  │
  │  - 文档加载（PDF/DOCX/TXT）                  │
  │  - 章节结构识别                              │
  │  - 关键字段提取（主体、金额、日期等）         │
  │  - 输出：结构化合同对象                      │
  └────────────────────┬────────────────────────┘
                       |
  ┌────────────────────▼────────────────────────┐
  │  风险扫描 Agent                              │
  │  - 语义检索规则库（RAG）                     │
  │  - 逐条款风险评级（低/中/高）                │
  │  - 标注异常条款位置与原因                    │
  │  - 输出：风险条款列表（结构化）              │
  └────────────────────┬────────────────────────┘
                       |
              ┌────────▼────────┐
              │  路由决策节点    │
              │ (Conditional Edge)|
              └──┬──────────┬──┘
           有高风险           无高风险
              |                  |
  ┌───────────▼──────┐   ┌──────▼──────────────┐
  │  人工审核暂停节点  │   │  报告生成 Agent      │
  │  (LangGraph       │   │  - 生成初审报告       │
  │   interrupt)      │   │  - 附修改建议         │
  │  - 推送给法务人员  │   │  - 输出 PDF/JSON 报告 │
  │  - 等待 approve/  │   └─────────────────────┘
  │    edit/reject    │
  └───────────────────┘
           |
  ┌────────▼────────────────────────────────────┐
  │  报告生成 Agent（含人工批注）                 │
  │  - 合并 Agent 分析与人工意见                 │
  │  - 生成最终审核报告（含审批记录）             │
  └─────────────────────────────────────────────┘
```

### 4.2 核心流程步骤详解

**Step 1：文档接入与预处理**

系统通过标准上传接口接收合同文档（支持 PDF、Word、扫描件 OCR 转文本）。LangChain 的 `PyPDFLoader` 或 `Docx2txtLoader` 将文档按页/段落切分为 `Document` 对象序列。对于长合同（超过 LLM 上下文窗口），采用递归字符切分策略（`RecursiveCharacterTextSplitter`），保留段落语义完整性。

**Step 2：结构化信息抽取**

解析 Agent 以 LLM + Structured Output 的方式，按预定义的 Pydantic Schema 提取合同关键字段：

```
ContractMetadata:
  - parties: List[Party]          # 合同主体列表
  - subject: str                  # 合同标的描述
  - total_amount: Optional[float] # 合同总金额
  - currency: str                 # 货币
  - effective_date: date          # 生效日期
  - expiry_date: Optional[date]   # 到期日期
  - governing_law: str            # 适用法律
  - dispute_resolution: str       # 争议解决方式
```

**Step 3：风险条款扫描（RAG + LLM 联合检测）**

风险扫描 Agent 对每个合同段落执行两阶段检测：

- **第一阶段（规则库匹配）**：将段落文本向量化，与存储在向量数据库中的风险规则库进行语义检索，返回 Top-K 相关规则。此步骤捕获已知风险模式。
- **第二阶段（LLM 推理）**：将原始段落 + 检索到的规则作为上下文，由 LLM 判断该段落的风险性质、风险等级（低/中/高）、风险原因，并给出修改建议。此步骤捕获规则库未覆盖的新型风险。

两阶段结合的优势：规则库提供稳定、可审计的已知风险基准；LLM 提供对新型条款的泛化推理能力。

**Step 4：条件路由——人工介入决策**

LangGraph 的条件边（`add_conditional_edges`）根据 State 中的风险扫描结果执行路由：

- 存在"高风险"条款 → 路由至人工审核暂停节点（`interrupt`）
- 仅有"中低风险"条款 → 路由至自动报告生成节点

暂停节点通过 LangGraph 的持久化层将当前 State 保存，并向法务人员推送审核任务通知（含 Agent 的分析摘要）。法务人员通过审核界面执行 approve/edit/reject 操作，工作流从暂停点恢复。

**Step 5：报告生成与归档**

报告生成 Agent 将结构化审核数据输出为正式报告，包含：

- 合同基本信息摘要
- 风险条款清单（含风险等级、原文位置、风险说明、修改建议）
- 人工批注记录（若有）
- 审核结论（通过/有条件通过/拒绝）
- 审核时间戳与审核人签名（可对接电子签章系统）

### 4.3 工具链配置

| 组件 | 推荐工具/方案 | 用途 |
|------|---------------|------|
| 文档加载 | LangChain `PyPDFLoader`、`Docx2txtLoader` | 合同文档解析 |
| 文本切分 | LangChain `RecursiveCharacterTextSplitter` | 长文档分块 |
| Embedding 模型 | OpenAI `text-embedding-3-large` 或私有化部署 | 合同语义向量化 |
| 向量数据库 | Chroma（开发）/ pgvector（生产） | 规则库与历史案例存储 |
| LLM | GPT-4.1 / Claude Sonnet（敏感文档优先私有部署） | 条款理解与风险推理 |
| Agent 框架 | LangChain `create_agent` + LangGraph `StateGraph` | 工作流编排 |
| 持久化 | LangGraph `PostgresSaver`（生产环境） | 审核状态持久化 |
| 人机交互 | LangGraph `interrupt` + `HumanInTheLoopMiddleware` | 人工审核节点 |

### 4.4 典型交互示例

以下是一次完整审核任务的交互流转示意：

```
[法务助手系统]
> 已接收合同文件：《XX 科技有限公司 SaaS 服务协议 v2.1.pdf》
> 正在解析文档结构...（解析 Agent 运行中）
> 已提取关键信息：
  - 甲方：XX 科技有限公司
  - 乙方：YY 集团采购部
  - 合同金额：人民币 580 万元
  - 服务期限：2026-04-01 至 2028-03-31
  - 争议解决：上海仲裁委员会仲裁
```

### 4.5 预期效益与局限性说明

**预期效益**

| 指标 | 预期改善 |
|------|----------|
| 单份合同初审时间 | 从 2-4 小时缩短至 5-15 分钟（Agent 自动初筛） |
| 法务人员有效工时占比 | 从 30%（大量时间在阅读）提升至 70%（集中在判断与谈判） |
| 风险条款漏检率 | 通过规则库 + LLM 双层检测，预期降低 60% 以上 |
| 审核记录可追溯性 | 全流程状态持久化，每步操作均有时间戳记录 |

**已知局限性与风险**

- **LLM 幻觉风险**：LLM 可能对某些条款给出错误的法律解释。缓解措施：所有高风险结论必须经人工确认，不允许 Agent 自主作出最终法律判断。
- **规则库时效性**：法律法规持续更新，规则库需要定期维护。缓解措施：建立规则库版本管理机制，法律变更后及时更新。
- **敏感数据合规**：合同通常包含商业机密，使用云端 LLM API 存在数据泄露风险。缓解措施：强监管行业建议私有化部署 LLM，或采用数据脱敏后再调用 API 的方式。
- **跨语言合同支持**：涉及境外合同（英文、中英双语）时，LLM 的中文法律知识可能覆盖不足。缓解措施：针对跨境合同场景单独配置具有相关法律知识的专用模型或 Prompt。
