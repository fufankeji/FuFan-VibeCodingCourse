## 二、各竞品文档审核流程详细拆解

### 2.1 Ironclad

**产品定位**：Ironclad 是面向企业法务团队的全生命周期合同管理（CLM）平台，2025年被 Gartner 列为 CLM 魔力象限领导者。核心亮点是"AI 运行在每个合同工作流步骤之后"，2025年8月新增 Jurist AI Agent，实现全合同 AI 初始红线标注（Redlining）。

**文档上传环节**

- 支持格式：PDF、Word（DOCX）、扫描件（OCR 支持）
- 批量上传：支持单次最多 100 份文档批量导入（SmartImport）
- OCR 能力：内置 OCR 扫描历史合同，自动 OCR 识别并索引遗留合同库
- 版本管理：支持合同谈判过程中的多版本上传、重命名、旧版本删除；支持并排查看版本差异与红线标注（Redline）
- 局限：SmartImport 元数据提取准确率存在约 20% 错误率，需人工校正

**文档解析处理**

- 通过 OCR 自动识别并标记 194+ 项合同元数据属性（AI-detected properties）
- AI Intake Agent 从第三方合同中自动提取关键元数据，辅助填充 Ironclad 系统的发起表单（Launch Form）
- 支持条款级别（clause-level）的结构化提取，用于规则触发
- 多语言支持未明确公开，主要针对英文合同优化

**AI 审核环节**

- Jurist AI Agent（2025年8月上线）：上传合同后自动执行全文档初始红线标注，并提供每条红线的推理说明；长合同提供摘要视图以降低审阅疲劳
- 触发方式：文档上传后自动触发，无需手动发起
- 审核维度：风险条款识别、与公司内部 playbook 的合规性对比、格式问题
- 规则引擎与 LLM 结合：可基于法务团队定义的 playbook 规则，由 AI 对偏差条款进行推理和标注；支持无代码 AI 训练，允许法务人员直接调整模型行为
- 审核时效：单份合同 AI 初审几分钟内完成（Jurist 初始红线标注）

**人工审核环节**

- HITL 介入时机：工作流可配置，通常在 AI 标注完成后触发人工审阅节点；高价值合同或特定触发条件（合同金额阈值、合同类型）强制路由至指定审批人
- 审核界面：Ironclad Editor（浏览器端原生编辑器）+ Microsoft Word 集成插件（Ironclad for Word）
- 批注工具：支持在线批注、红线标注、评论留痕
- 审批流程：拖拽式无代码工作流编辑器（Workflow Builder），可配置多级审批链、基于风险等级/合同金额/合同类型的条件路由
- 多人协作：支持多审阅人同时标注，合同状态对所有协作方实时可见

**流程编排**

- 状态流转：草稿→审阅中→谈判→签署→执行中→归档，全程状态可见
- 路由规则：基于合同价值阈值、风险等级、合同类型自动路由至不同审批链；支持 Salesforce、Slack 等系统集成触发通知
- 审核链路追溯：完整操作审计日志，记录每步操作时间戳与操作人；与电子签章（DocuSign 等）集成
- 自动通过规则：低风险/标准模板合同可配置自动通过无需人工；高价值合同强制多级审批

### 2.2 Kira Systems（Litera Kira）

**产品定位**：Kira 是全球领先的合同智能提取与分析平台，深度服务于顶级律所（占据全球 Top 100 律所约 70%，Top 25 并购律所 80%+）。核心能力是 1,400+ 预训练条款字段的精准提取，以及支持高并发大体量文档项目的批量处理。2025年7月 Litera 推出新 Kira 体验，新增 Generative AI Smart Fields（生成式智能字段）。

**文档上传环节**

- 支持格式：PDF（含数字 PDF 与扫描件）、Word（DOCX）、数据房（Data Room）直接集成（Intralinks、Merrill、HighQ）
- 批量上传：支持大规模批量导入（Matter/Project 级别，单项目支持数万文档），内置 Keep-Awake 机制防止长时间批量上传超时；支持去重（Deduplication）
- 版本管理：支持文档版本控制，配合数据房集成可直接拉取最新版本
- 预处理：上传时自动识别 OCR 质量并标记低质量扫描件，支持 100+ 语言自动识别

**文档解析处理**

- OCR 能力：内置 OCR 处理，自动识别 OCR 质量等级；对低质量扫描件单独标记，由 Matter Manager 决定是否获取更高质量原件
- 结构化提取：1,400+ 预训练"Smart Fields"（智能字段），覆盖 40+ 实质性法律领域（并购、租赁、贷款、劳工等）；支持用户自训练无限数量的自定义字段，无需代码
- Generative Smart Fields（2025年新增）：通过自然语言 Prompt 快速创建新字段，支持任意语言，无需传统 ML 训练数据
- Concept Search：通过单个示例概念跨文档全库语义搜索
- 多语言支持：自动识别语言，支持全球主流语言

**AI 审核环节**

- 触发方式：文档上传后自动触发字段提取与条款识别；用户可配置审查规则自动标注
- 审核维度：精准的条款存在/不存在检测、条款内容提取与分类、风险标注（Flags/Tags 系统）
- 规则引擎与 LLM 结合：传统 ML 训练模型（高精度已知字段）+ GenAI（新字段快速创建，降低配置门槛）；优先保证已知字段的可靠性，GenAI 用于扩展覆盖边界
- 审核时效：批量提取效率高，大型并购 Due Diligence 项目数千份文件可在数小时内完成初步提取

**人工审核环节**

- HITL 介入时机：提取完成后，审阅人在"Analysis Grid"（分析网格视图）中对结果进行验证与纠错；Matter Manager 可按复杂度/专业领域将文件批量分配给不同审阅人
- 审核界面：Analysis Grid（表格式批量审阅视图，可跨文档对比同类条款）+ Document Viewer（单文档阅读视图，AI 标注高亮显示）
- 批注工具：Flags/Tags 标记系统，支持协作留痕；支持对字段提取结果进行人工确认/修改
- 审批流程：Matter Manager 仪表盘追踪整体进度，可实时看到每位审阅人的完成量；支持将审阅结果导出为 Word/Excel/PDF 报告
- 多人协作：专为多人协作大项目设计，支持任务分配与进度追踪

**流程编排**

- 状态流转：文档导入→OCR 处理→字段提取→人工审阅→导出报告
- 路由规则：按文档类型、复杂度、语言自动分组并分配审阅任务
- 审核链路追溯：所有提取结果均链接到原始文档源语言（Source Language），可一键跳转验证；操作日志记录审阅人修改历史
- 局限：Kira 侧重提取与分析，审批工作流和状态管理能力相对较弱（通常配合其他 CLM 平台使用）

### 2.3 Luminance

**产品定位**：Luminance 是英国 Legal-Grade AI 公司，基于自研法律 LLM（训练语料超过 1.5 亿份已验证法律文档），覆盖合同谈判、尽职调查、CLM 全生命周期。核心产品包括：Negotiate（Word 插件，谈判中实时审查）、Due Diligence（批量尽调）、CLM（全生命周期管理）。

**文档上传环节**

- 支持格式：PDF、Word（DOCX）、扫描件（OCR）；支持主流数据房和文档管理系统集成
- 批量上传：支持大批量文档项目，尽调场景支持跨文件夹批量处理
- 版本管理：支持合同版本管理，谈判过程中实时追踪版本变化
- 预处理：OCR 识别，多语言自动检测

**文档解析处理**

- OCR 能力：内置 OCR，支持扫描件处理
- 结构化提取：基于自研法律 LLM 的条款级语义理解，自动识别并分类合同条款（终止条款、保密条款、赔偿条款等）
- 多语言支持：支持多语言合同分析（具体语言范围公司官网标注为"多国语言"，未公开完整列表）
- Playbook 支持：与公司内部 Playbook 对比，自动标注偏差条款；Auto-Markup 功能自动生成合规条款建议替换

**AI 审核环节**

- Negotiate 模块（Word 插件）：实时检测合同中与标准 Playbook 偏差的条款并高亮提示，提供合规替换建议
- Due Diligence 模块：批量分析模式，类似 Kira 的批量字段提取
- 触发方式：Word 插件实时触发（Negotiate）；文档上传后批量触发（Due Diligence）
- 审核维度：风险条款（非标准条款、偏离 Playbook 条款）、关键义务、关键日期
- 规则引擎与 LLM 结合：自研法律 LLM 为核心推理引擎，Playbook 规则作为约束层
- 审核时效：Negotiate 模式为实时（秒级）；批量尽调模式为数分钟至数十分钟

**人工审核环节**

- HITL 介入时机：AI 标注完成后法律团队在 Word 插件内直接处理建议（接受/修改/忽略）；高风险条款优先级排序引导人工关注
- 审核界面：Microsoft Word 插件（Negotiate）+ 浏览器 Dashboard（CLM 模块）
- 批注工具：Word 原生批注 + Luminance 插件内嵌建议卡片；支持批注留痕
- 审批流程：拖拽式 Workflow Editor（Drag-and-Drop），可配置基于合同属性（内容、适用法律、合同类型、语言）的自动路由与多级审批
- 多人协作：合规模块允许非法律团队（销售、HR）直接从预批准模板生成合同，法律团队专注高价值审查工作

**流程编排**

- 状态流转：草稿/导入→AI 分析→法律审阅→审批→签署→归档
- 路由规则：基于合同内容属性、风险等级、管辖法律自动路由；低风险合同可走快速通道
- 审核链路追溯：完整审计日志；Dashboard 提供整体合同数据与审阅进度可视化
- 差异化：自研法律 LLM 是核心竞争壁垒，非通用 LLM 调用，对法律语义理解更精准

### 2.4 Harvey AI

**产品定位**：Harvey 是面向顶级律所和企业法律团队的专业 LLM 平台，使用 OpenAI 技术深度定制。2025年推出端到端 M&A 法律工作流平台，核心产品包括：Vault（文档存储与管理）、Review Tables（批量文档提取与分析）、Workflows（可定制自动化工作流）、Word 插件。

**文档上传环节**

- 支持格式：PDF、Word（DOCX）
- 批量上传：Vault 文档系统支持大规模文档上传，单个 M&A 项目可处理 50,000+ 文档，系统架构支持 100,000+ 文档的单项目规模
- 版本管理：Vault 支持文档组织管理，文档分层结构（文件夹/标签）
- 预处理：文档上传后系统智能推荐最适合的处理方式（Agentic 分析）

**文档解析处理**

- 结构化提取：Review Tables 功能——可跨文档集自定义提取数据点，每个提取结果均链接到原始源语言，支持全溯源验证
- 多语言支持：依托底层 LLM 能力，支持多语言文档（非公开具体语言范围）
- Agentic 文档分析：自动推荐文档处理方式，Agent 驱动的 Due Diligence 报告生成（PwC 合作案例：单个工作流执行超 10,000 次）

**AI 审核环节**

- Workflows（工作流引擎）：用户可运行预构建工作流，或自定义工作流嵌入公司专有知识
- 触发方式：文档上传后可手动触发工作流，也支持批量触发
- 审核维度：条款提取、风险识别、M&A 交割清单生成、合规检查；M&A 场景可自动生成签约/交割/交割后清单
- 规则引擎与 LLM 结合：用户将公司专有知识（内部规则、标准）编码进工作流；LLM 负责推理执行；2025年 Harvey 用户已构建 18,000+ 自定义工作流
- 审核时效：单文档分析秒级至分钟级；大批量项目为小时级

**人工审核环节**

- HITL 介入时机：Review Tables 中支持将行分配给特定用户（任务分配）；支持用"红色标记"（Flag）标注需关注的条目
- 审核界面：浏览器端 Review Tables（表格视图）+ Word 插件（文档编辑视图）；Outlook 插件（2025年8月新增，支持邮件内直接使用 Harvey）
- 批注工具：表格内可对单元格加红旗标记（Flag）；支持验证/未验证状态筛选
- 审批流程：面向律所/法律团队的工作流编排，支持在现有工作流中插入审阅步骤；Word 插件支持上下文感知的修改建议
- 多人协作：Review Tables 支持按用户分配审阅行，团队协作拆分文档集

**流程编排**

- 状态流转：文档导入（Vault）→Workflow 触发→AI 分析（Review Table 或 Agentic 报告）→人工审阅/标注→导出/交付
- 路由规则：工作流步骤中可插入条件节点，支持逻辑路由
- 审核链路追溯：每个提取结果均可溯源到原始文本；Word 插件编辑历史记录
- 差异化：Harvey 作为"工作流+文档智能"平台，用户可将公司专有知识编码为可复用工作流，形成组织级知识资产

### 2.5 Spellbook

**产品定位**：Spellbook 是深度集成在 Microsoft Word 内的 AI 合同审查与起草工具，适合律师和法务在原有工作环境中无缝使用。核心能力是实时风险标注、AI 建议红线，以及对比 2,000+ 行业 Benchmark 的合规性检测。

**文档上传环节**

- 支持格式：Word（DOCX）——Spellbook 完全工作在 Word 环境内，不支持独立平台文档上传
- 批量上传：不支持独立平台批量上传；Word 内单文档操作模式
- 版本管理：依赖 Word 原生版本管理和 Track Changes

**文档解析处理**

- Word 内实时解析：安装 Word 插件后，打开合同文档即自动触发 AI 分析
- 结构化提取：自动识别条款结构，对比 2,000+ 行业 Benchmark
- 多语言支持：依托底层 LLM，支持多语言（未公开完整列表）
- 无需独立文档上传流程：文档完全保留在用户本地 Word 环境中

**AI 审核环节**

- 触发方式：在 Word 内打开合同并启动 Spellbook 面板后，点击"运行 AI 审查"
- 审核维度：客户风险识别（Risk Scanning）、起草错误检测、Benchmark 对比、缺失条款提示（Missing Clause）、自定义 Playbook 对比
- 规则引擎与 LLM 结合：2,000+ 行业 Benchmark 库作为规则层，LLM 负责语义理解与建议生成
- 审核时效：秒级响应（Word 插件内实时，不涉及异步处理）
- 学习能力：Spellbook 可"记住"用户偏好，逐步优化针对特定用户立场（甲方/乙方）的建议

**人工审核环节**

- HITL 模式：Word 原生模式——AI 建议以 Tracked Changes（追踪修改）和 Comments（批注）形式插入文档，用户直接在 Word 中接受/拒绝/修改
- 审核界面：Microsoft Word（无独立审核界面）

## 四、关键发现与差异化机会

### 4.1 跨竞品普遍存在的设计模式

**发现 1：AI 初审 + 人工精审成为行业标配**

所有 10 家竞品均采用"AI 先跑、人工后确认"的双层架构，无一产品声称完全自动化审核。这印证了 01_business_research 阶段的核心结论：人机协作是刚性需求。差异体现在 HITL 触发条件的精细化程度——成熟平台（Ironclad、Luminance、ContractPodAi）支持基于风险等级、合同金额、合同类型的条件触发，而工具层产品（Spellbook、Robin AI 早期版本）HITL 机制较为简单。

**发现 2：Microsoft Word 是人工审核的核心战场**

Luminance、Spellbook、Robin AI、Harvey AI 均提供 Word 插件，将 AI 能力注入法律人员的原有工作环境（而非强迫其切换到新界面）。这一设计显著降低了用户摩擦。Word 插件已成为面向律所和企业法务的产品标配，而非差异化优势。

**发现 3：Playbook/规则库是 AI 审核的"锚点"**

所有平台均提供某种形式的公司内部规则/标准配置层（Playbook、Smart Fields、Standard Positions、审查规则库），AI LLM 在此规则锚点约束下运行，而非完全自由推理。这大幅提升了输出结果的一致性与可审计性。

**发现 4：批量大项目处理是律所/并购场景的核心诉求**

Kira 和 Harvey AI 在大规模文档处理（单项目 5,000~100,000 份）上具有明显设计优势（项目级批量导入、Review Table 的表格式批量提取、Matter Manager 的任务分配仪表盘）。这是企业法务日常合同审查与并购尽调两类场景分野的关键维度。

**发现 5：国内产品 OCR 能力更强，但流程编排成熟度相对较低**

达观数据在 OCR 精度（99%+）和文档格式覆盖广度（20 种+）上处于明显领先地位，是扫描件/复杂格式文档处理的重要技术优势。幂律智能 MeCheck 在中文合同场景的审查规则库深度方面有积累。但相比 Ironclad/Luminance/ContractPodAi 等平台，国内产品的工作流编排（状态机、条件路由、审计日志完整性）尚未达到同等成熟度。

### 4.2 主要差异化机会

**机会 1：LangGraph 状态机与 HITL 的工程级实现**

目前市场上的竞品大多将 HITL 实现为"工作流审批节点"（基于 no-code 配置），底层是传统状态机或 BPM 引擎，缺乏对**异步跨天暂停-恢复**的原生支持。LangGraph 的 `interrupt/resume` + `Checkpointer` 持久化机制提供了一种更底层、更可靠的 HITL 工程实现：
- 支持真正的异步跨天恢复，法务人员一周后响应也不会丢失状态
- 每步 State 快照使审核链路完全可追溯
- Conditional Edge 路由逻辑可表达复杂的多维度风险路由规则

这是本项目相比市场上"配置式 HITL"的核心技术差异化点。

**机会 2：多 Agent 分工架构的透明化**

现有竞品（包括 ContractPodAi 的 Leah Agentic OS）开始引入"Agentic"概念，但大多数产品对 AI 内部执行过程不透明（用户只看到输入/输出，不知道 AI 做了哪些子步骤）。本项目采用 LangGraph 显式 Graph 架构（解析 Agent → 风险扫描 Agent → 路由节点 → 人工节点 → 报告 Agent），每个节点的执行状态在系统中清晰记录，具备对用户展示"AI 正在做什么"的技术条件，提升可解释性。

**机会 3：RAG 增强的规则库检索**

多数竞品使用静态规则库（固定字段列表、固定 Benchmark 对比），对规则库的检索是精确匹配而非语义检索。本项目采用 LangChain RAG（向量数据库 + 语义检索）的方式访问规则库，在遇到新型条款时能通过语义相似性找到最相关的既有规则片段，比静态规则库更具泛化能力。这在国内产品中几乎是空白（幂律智能是最接近的竞争者，但其公开信息显示以规则库精确匹配为主）。

**机会 4：中文合同全链路的深度优化**

Kira 和 Luminance 的多语言支持主要依赖通用 LLM，对中国法律体系（个人信息保护法、数据安全法、劳动合同法等）的专项理解有限。达观数据和幂律智能在中文规则库上有积累，但工程化程度（工作流编排、审计日志、API 集成）相对薄弱。本项目面向中文合同场景，可以在规则库的中国法规覆盖深度和工程化流程编排之间取得优化平衡。

**机会 5：分级审核策略的产品化表达**

目前市场上大多数产品的"低风险自动通过"规则较为粗糙（基于简单阈值），缺乏对"中风险条款"处理方式的精细分层（批量待复核 vs 即时推送 vs 系统记录但不打断）。本项目可以在 LangGraph Conditional Edge 中表达更丰富的路由策略，形成三档分级（低风险自动通过→中风险批量复核→高风险强制打断），并以产品化方式让用户配置分级阈值。

## 五、对本项目流程设计的建议

基于以上竞品分析，结合 01_business_research 阶段的结论，提出如下设计建议：

### 建议 1：明确产品层级定位，避免与 CLM 平台正面竞争

市场上 Ironclad、ContractPodAi（Leah）等已是成熟的全生命周期 CLM 平台，覆盖从合同起草到归档的所有环节。本项目基于 LangGraph 构建，应定位为**"合同审核引擎"**（聚焦上传→解析→AI审核→HITL→报告生成这个核心审核链路），而非试图复制完整 CLM 平台。核心竞争力在于：工程级可靠的 HITL 机制 + 透明可解释的多 Agent 执行过程 + 深度中文法律场景优化。

### 建议 2：文档上传环节优先支持 PDF 和 DOCX，OCR 采用成熟方案

- PDF（数字 PDF）+ Word（DOCX）是 10 家竞品全部支持的最低覆盖范围，本项目必须支持
- 扫描件 OCR 建议接入达观 TextIn 或合合信息 OCR API（而非自研），99%+ 精度是扫描件场景的生产要求
- LangChain 的 `PyPDFLoader`、`Docx2txtLoader` 已能满足数字 PDF 和 Word 加载需求
- 批量上传作为进阶功能，MVP 阶段可先实现单文档，再扩展为批量队列

### 建议 3：AI 审核触发应为自动触发，并向用户展示执行进度

- 竞品主流模式为文档上传后自动触发 AI 审核，本项目应保持一致
- LangGraph 的显式 Graph 执行使得向用户展示"当前执行到哪个 Agent 节点"成为可能，应在 UI 中实现执行进度展示（"正在解析文档结构..." → "正在扫描风险条款..." → "等待法务人员确认..."）
- 这一能力在竞品中几乎缺失，是透明化 AI 执行的差异化体验

### 建议 4：HITL 节点实现三类决策（Approve/Edit/Reject），并记录完整决策上下文

- 对标 Ironclad 和 ContractPodAi 的接受/修改/拒绝模型，实现 LangGraph `interrupt` 后的三类恢复路径
- Edit 决策时，人工修改内容应与 Agent 原始分析结果一并存入 State，最终审核报告需包含"Agent 初始判断"和"人工修改意见"的对比记录
- 这种决策上下文的完整记录在竞品中普遍缺失，是审计合规场景的重要差异化

### 建议 5：风险路由规则配置化，支持三档分级

| 风险档位 | 路由行为 | 对应 LangGraph 实现 |
|----------|----------|---------------------|
| 低风险 | 自动通过，生成报告，不打断法务工作流 | Conditional Edge 路由至报告生成节点 |
| 中风险 | 记录待复核列表，批量推送给法务（不打断当前审阅） | 加入 State 的待复核队列，异步通知 |
| 高风险 | 立即 interrupt，强制等待法务确认后才继续 | LangGraph `interrupt()` 调用 |

分级阈值应可由用户配置（不同合同类型/金额段使用不同阈值），这是本项目 MVP 阶段的核心产品能力。

### 建议 6：审核报告包含完整的 Agent 执行溯源信息

参考 Harvey AI 的"每个提取结果均链接到原始源语言"的做法，本项目的审核报告中每个风险条款应包含：
- 原文位置（章节 + 页码）
- Agent 的推理依据（触发了哪条规则库规则、LLM 给出的推理说明）
- 风险等级与判断依据
- 人工处理结论（若有 HITL）

LangGraph Checkpointer 使每步执行记录均可持久化，为报告溯源提供完整数据基础。

### 建议 7：初期不实现 Word 插件，通过 Web 界面完成 HITL

- Spellbook、Robin AI、Luminance 的 Word 插件是面向"在原有工作环境中不切换工具"诉求的产品决策，开发和维护成本较高
- 本项目 MVP 阶段以 Web 界面完成 HITL（文档预览 + 风险条款标注 + Approve/Edit/Reject 操作）
- Word 插件作为后续阶段的扩展选项
