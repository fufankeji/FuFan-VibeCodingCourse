## 二、双维度交叉发现

以下为两份报告的核心发现在交叉比对后提炼的共识结论：

### 共识 1：AI 初审 + 人工精审是行业刚性标配

- **流程维度**（T1）：10 家竞品均采用"AI 先跑、人工后确认"双层架构，无一声称全自动化
- **呈现维度**（T2）：所有产品均提供某种形式的"接受/拒绝/修改"操作界面，人工决策是流程闭环的必经环节
- **结论**：印证 01 阶段核心共识——人机协作是刚性需求

### 共识 2：Playbook/规则库是 AI 审核的"锚点"

- **流程维度**（T1）：所有平台都提供公司内部规则/标准配置层，LLM 在规则约束下运行
- **呈现维度**（T2）：规则引擎结果与 AI 判断在界面上区分展示（达观数据的双层标注为最佳实践）
- **结论**：本项目应以 LangChain RAG + 规则库作为 AI 审核的约束层，且在结果呈现中明确区分"规则触发"与"AI 推理"两种来源

### 共识 3：原文关联是审核体验的核心

- **流程维度**（T1）：Kira 的 Analysis Grid + Document Viewer 双栏架构是大规模提取场景的标杆
- **呈现维度**（T2）：Kira 的双向关联双栏对照被评为原文定位的行业最佳实践
- **结论**：本项目必须实现"结果 ↔ 原文"的双向关联，MVP 阶段即需包含此能力

### 共识 4：置信度/不确定性展示是行业共识

- **流程维度**（T1）：Kira 对低质量 OCR 文档单独标记，Ironclad SmartImport 标注元数据错误率
- **呈现维度**（T2）：Kira 的置信度分数（0-100%）和 Harvey AI 的不确定性声明是两种代表模式
- **结论**：本项目所有 AI 判断必须附带不确定性标识，直接对应 01 阶段反方报告"LLM 幻觉与过度自信"风险

### 共识 5：国内产品法规引用深度领先，但流程编排成熟度不足

- **流程维度**（T1）：达观数据 OCR 精度 99%+，但工作流编排低于 Ironclad/Luminance 等国际平台
- **呈现维度**（T2）：幂律智能法条引用驱动模式是国内场景参考标杆
- **结论**：本项目可在"法规引用深度 + LangGraph 工程级流程编排"两个维度同时发力，形成国内外竞品均不具备的组合优势

### 共识 6：LangGraph 特性是本项目的核心差异化机会

- **流程维度**（T1）：`interrupt/resume` 异步跨天恢复在竞品中几乎无对标
- **呈现维度**（T2）：工作流状态可见性（当前节点、已完成步骤、checkpointer 状态）是所有竞品均不具备的呈现维度
- **结论**：LangGraph 的 StateGraph 可视化 + 异步 HITL 恢复是本项目最强差异化点，应在产品层面显性化

## 五、设计禁区汇总

综合两份报告，以下设计模式应明确避免：

| 禁区 | 风险来源 | 替代方案 |
|------|----------|----------|
| 不说明审核覆盖范围，仅展示"审核完成" | 信息损失（T2） | 显示已覆盖和未覆盖的条款类型 |
| 高风险确认仅需单击（无摩擦） | Automation Bias（T2） | 强制展开原文 + 填写接受原因 |
| 以单一置信度分数呈现所有结果 | LLM 过度自信（T2） | 不同类型判断用不同不确定性表达 |
| 审核报告使用"AI 已全面审核"等表述 | 法律责任风险（T2） | 明确标注"AI 辅助分析，最终判断由人工负责" |
| 试图复制完整 CLM 平台 | 与 Ironclad 等正面竞争（T1） | 聚焦审核引擎核心链路 |
| 自研 OCR 引擎 | 工程投入巨大、精度难保证（T1） | 接入达观 TextIn 等成熟 OCR API |
