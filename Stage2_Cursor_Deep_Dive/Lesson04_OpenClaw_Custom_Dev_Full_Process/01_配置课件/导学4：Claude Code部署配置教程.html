1. 什么是 Claude Code

在正式进入安装配置之前，我们有必要先建立对 Claude Code 的深层认知。很多同学可能已经听说过它，甚至已经在使用，但对它的技术本质、底层架构和工作机制未必有系统的理解。这一章的目标，不仅是帮助大家从「听说过」升级到「真正懂」，更要从架构层面回答一个关键问题：Claude Code 为什么能在 SWE-bench Verified 测试中达到 80.9% 的自主问题解决率，远超多数竞品？

要回答这个问题，我们需要理解一个核心概念的转变：Claude Code 并非单纯的代码生成器，也不是嵌入 IDE 的聊天窗口。用更精确的术语来说，它是一个代理套件（Agentic Harness）——将 Claude 模型的推理能力封装在一个能够直接访问文件系统、运行终端命令并集成版本控制系统的执行环境中。它的工作模式基于「代理循环（Agentic Loop）」，涵盖上下文采集、行动执行与结果验证三个交织的阶段。这意味着它不仅能写出代码，还能通过运行 npm test 或 pytest 来验证代码的正确性，并根据报错信息自主进行多轮重构，直到任务达成预定目标。这种端到端的任务处理能力，使其在处理诸如「修复整个项目的 Lint 错误」或「为身份验证模块编写完整测试用例」等复杂任务时表现卓越。

1.1 产品定位与核心理念

用官方文档的原话来说，Claude Code 是一个「可以读取你的代码库、编辑文件、运行命令，并与你的开发工具集成」的代理编码工具（Agentic Coding Tool）。它可以在终端（Terminal）、IDE（VS Code / JetBrains）、桌面应用和浏览器中使用。

这里需要特别理解的一个设计哲学是：Claude Code 坚持「终端优先」的 Unix 哲学。它认为终端是开发者的核心指挥部，通过 CLI 接入可以实现与编译、构建及测试工具链的零距离对接。这种设计减少了 UI 层面的抽象带来的认知损耗，使得智能体能更直接地感知执行环境。在终端中输入 claude 即可启动一个交互式会话，你用自然语言描述你想做的事情，Claude Code 会自主规划执行路径、读取相关文件、编写代码、运行测试、提交 Git 变更——整个过程就像你在和一个经验丰富的同事进行结对编程（Pair Programming）。

与传统插件的本质区别在于：传统 AI 编程工具是开发者思维的延伸，而 Claude Code 是一个具备独立行动能力的数字同事。它不需要你逐行告诉它做什么，只需要给出目标和约束条件，它就会自行规划路径并执行。Anthropic 内部目前 100% 的代码都经过了 AI 生产或辅助，工程师的角色正在向「AI 团队管理者」转变——这不是未来的预测，而是正在发生的现实。

1.2 三层核心架构：模型、工具与环境的协同

理解了 Claude Code 的定位之后，一个自然的问题是：它的效能从何而来？答案在于其独特的三层协作架构。这三层各司其职、紧密协同，构成了 Claude Code 区别于其他 AI 编程工具的技术根基：

模型层（The Brain）：这是 Claude Code 的「大脑」，主要采用 Claude Sonnet 或更高级别的 Opus 模型。这些模型经过专门微调，在代码推理、长序列规划和工具调用准确率方面处于行业领先地位。在 SWE-bench Verified 测试中，其自主解决问题的成功率达到了 80.9%。2026 年版本中进一步引入了「Extended Thinking」机制——模型在行动前会分配数千个 Token 进行纯推理，模拟各种可能的边界情况，从而大幅降低因逻辑疏忽导致的回归 Bug。

工具层（The Hands）：这是 Claude Code 的「双手」，为模型提供了一套标准化的操作接口。通过这些接口，模型可以执行文件读写（read_file、edit_file）、正则搜索（grep）、目录扫描（ls）、终端交互（bash）以及多文件同步编辑（MultiEdit）等操作。这些工具不是花哨的附加功能，而是让 AI 从「只能说」进化到「能动手」的关键桥梁。

环境层（The Field）：这是 Claude Code 的「战场」，智能体直接运行在开发者的本地工作目录中。它能够感知当前的 Git 状态、环境变量、项目配置文件（如 package.json、pyproject.toml）以及通过 CLAUDE.md 文件传达的团队规范。这种深度的环境感知能力，使得 Claude Code 的输出天然贴合项目的实际技术栈和编码风格。

核心洞察：正是「模型的推理能力 + 工具的执行能力 + 环境的感知能力」三者的紧密协同，使得 Claude Code 能够完成从理解需求到验证交付的端到端闭环，而不仅仅是生成一段代码片段。

1.3 代理循环（Agentic Loop）：为什么 Claude Code 如此高效

三层架构提供了「能力基座」，而让 Claude Code 真正高效运转的核心引擎，是其代理循环（Agentic Loop）机制。与传统的「用户提问 → AI 回答」单轮交互不同，Claude Code 在接到任务后会进入一个自驱动的多轮循环，直到任务完成或需要人工决策时才停下来。

这个循环包含四个交织的阶段：

探索（Explore）：智能体首先调用 ls、grep 等工具扫描项目结构，理解代码库的组织方式、技术栈和依赖关系。这一阶段的目标是建立「项目地图」，为后续的精准操作奠定基础。

规划（Plan）：在充分探索的基础上，模型将复杂需求拆解为可执行的子任务（Subtasks）。对于大型任务，它会在 CLAUDE.md 或 SPEC.md 中记录执行计划，确保思路清晰、步骤有序。

实施（Implement）：利用 MultiEdit 等工具同时修改多个相关文件。例如，在修改 API 定义时，它会同步更新对应的 Mock 数据和测试用例，而不是只改一个文件留下一堆不一致。

验证（Verify）：运行 Shell 指令（如 npm test、pytest、cargo build）查看构建是否通过。如果失败，它会将 stdout/stderr 的报错信息输入回模型，自动开始第二轮循环——修改代码、重新测试——直到所有测试通过或达到重试上限。

正是这种「自我反馈」机制，使得 Claude Code 生成的代码 rework 率比竞品低约 30%——即它产出的代码更具生产就绪性，更少需要人类进行二次修正。研究表明，在处理「修复整个项目的 Lint 错误」或「为身份验证模块编写完整测试用例」等复杂任务时，代理循环的优势尤为显著。

1.6 安全与权限模型

赋予 AI 执行终端命令和编辑文件的能力，意味着我们必须认真对待安全边界的问题。Claude Code 在设计上采用了权限分级模型，确保开发者始终掌握控制权。理解这些权限级别，是安全使用 Claude Code 的前提：

Claude Code 权限分级模型

权限模式 行为说明 适用场景
default（标准模式） 写文件、运行 Bash 命令前必须获得用户逐一批准 日常开发，安全性最高
plan（计划模式） 只读模式，仅分析代码，不执行任何写操作 代码审查、架构分析
acceptEdits 代码编辑自动批准，但 Bash 命令仍需人工审核 信任编辑但需控制命令执行
全自动模式 所有操作自动批准（需谨慎） 受控的 CI/CD 环境

在底层实现上，Claude Code 通过操作系统级别的沙箱机制保护用户隐私和系统安全：在 Linux 上使用 bubblewrap，在 macOS 上使用 seatbelt，来限制文件读取范围和网络访问。严禁在未沙箱化的环境下使用 --dangerously-skip-permissions 参数——这会完全绕过安全检查，仅适用于完全受控的自动化流水线。

在交互操作层面，两个快捷键值得记住：按 Esc 两次可以紧急停止操作并触发自动快照回滚（Checkpointing），按 Shift+Tab 可以在 Normal（询问模式）、Auto-Accept（自动模式）和 Plan（计划模式）之间循环切换。

1.7 运行环境与使用方式

最后，我们来梳理 Claude Code 的多平台使用方式。它提供了五种入口，适配不同开发者的偏好和工作场景：

终端 CLI：最核心、功能最完整的使用方式。在终端中输入 claude 启动交互式会话，或使用 claude "task" 执行一次性任务。这也是本教程重点讲解的安装目标。

VS Code / Cursor 扩展：在编辑器中提供内联 Diff 视图、@-提及、计划审查等功能。安装方式是在扩展市场搜索 "Claude Code"。

JetBrains 插件：支持 IntelliJ IDEA、PyCharm、WebStorm 等 JetBrains 全家桶，提供交互式 Diff 查看和选择上下文共享。

桌面应用：独立应用程序，支持可视化 Diff 查看、多会话并行和云会话启动。支持 macOS 和 Windows。

Web 版：在浏览器中运行，无需本地安装。适合启动长时间运行的任务或处理本地没有的仓库。访问 claude.ai/code 即可使用。

重要提示：所有使用方式连接的是同一个底层 Claude Code 引擎，你的 CLAUDE.md 文件、设置和 MCP 服务器配置在所有界面中通用。你甚至可以用 /teleport 命令将 Web 上的会话拉入终端，或用 /desktop 将终端会话交接给桌面应用——这种跨界面的流动性，是 Claude Code 生态的一大亮点。

2. Claude Code 与 Cursor、Codex 的区别与联系

在上一章中，我们从架构内幕（三层协作）、工作机制（代理循环）、功能矩阵到安全模型，建立了对 Claude Code 的深层技术认知。接下来一个自然而然的问题是：它和市面上其他热门的 AI 编程工具——尤其是 Cursor 和 OpenAI Codex——到底有什么区别？各自适合什么场景？这是很多开发者在工具选型时最关心的问题。本章将基于 2026 年初的行业基准数据进行系统性横向对比，帮助大家建立清晰的认知框架。

需要先说明的是，这三款工具并不是简单的「竞品」关系，它们代表了 AI 编程辅助的三种不同范式：Cursor 是「IDE 优先」路线，Codex（2026 版桌面应用）定位为「代理指挥中心」，Claude Code 则坚持「终端优先」的 Unix 哲学。理解这种范式差异，比单纯比较功能列表更有价值。

2.2 架构与工作流差异深度分析

对比表给出了全景视图，但要真正理解这三款工具的差异，我们还需要深入它们的架构和工作流层面。

Claude Code 的「终端原生代理」模式：Claude Code 的设计哲学是「让 AI 生活在你的终端里」。当你在终端中输入 claude 启动会话后，AI 获得的是与你相同的开发环境访问权限——它可以读取你的文件系统、运行你的构建工具、操作你的 Git 仓库。这种设计意味着 Claude Code 不需要额外的索引步骤，它通过内置的 ripgrep 进行实时代码搜索，按需读取文件。更重要的是，它的每一个操作（编辑文件、运行命令）都需要你的明确批准，这在安全性和可控性上提供了很好的保障。

Cursor 的「IDE 增强」模式：Cursor 本质上是一个基于 VS Code 分叉的增强型 IDE。它的核心优势在于编辑器级别的深度集成：实时 Tab 补全、内联 Diff 显示、上下文感知的代码建议。对于习惯在图形化 IDE 中工作的开发者来说，Cursor 提供了最无缝的 AI 辅助体验。但它的代理能力（Agent Mode）相对较新，在自主执行复杂多步骤任务方面，不如 Claude Code 成熟。

Codex 的「云端沙箱」模式：OpenAI Codex 采用了完全不同的路径——它将代码执行完全放在云端沙箱中。当你给 Codex 一个任务时，它会将你的 GitHub 仓库拉取到云端环境中，在隔离的沙箱里完成代码修改和测试，最终以 Pull Request 的形式将结果推送回你的仓库。这种模式的好处是完全不影响你的本地环境，但缺点是你无法实时观察和干预 AI 的工作过程。

组合一：Claude Code + Cursor。这是目前最流行的组合方式。用 Cursor 进行日常的代码编辑和实时补全（快速写代码的效率无可匹敌），用 Claude Code 处理需要跨文件重构、复杂调试、Git 操作等需要「深度思考和自主执行」的任务。事实上，Claude Code 官方就提供了 VS Code 扩展，可以在 Cursor（基于 VS Code）中直接使用。

3.2 Windows 安装配置（图文全流程）

Windows 是三大平台中安装配置最复杂的，原因在于 Windows 不原生支持 Unix Shell，Claude Code 需要借助 Git Bash 作为命令执行环境。本节通过真实操作截图，带你从环境准备到成功启动，一步不落。

步骤一：安装 Claude Code（PowerShell 原生安装）

在 Windows 上安装 Claude Code，官方最推荐的方式是 PowerShell 原生安装，支持自动更新，安装后 claude.exe 位于 %USERPROFILE%\.local\bin\ 目录。

（1）以管理员身份打开 PowerShell

按 Win + S 搜索 "PowerShell"，在搜索结果中右键点击 "Windows PowerShell"，选择 "以管理员身份运行"：

在打开的管理员 PowerShell 窗口中，先确认 Git 是否已安装（Claude Code 需要 Git Bash 作为执行环境）：

git --version

如果输出了版本号（如 git version 2.x.x）说明 Git 已就绪，直接进入下一步。如果报错"命令不存在"，需要先安装 Git for Windows：

  1. 访问 https://git-scm.com/downloads/win 下载 Windows 版 Git
  2. 安装时勾选 "Git Bash Here" 和 "Add to PATH" 选项
  3. 安装完成后，重新打开 PowerShell 并再次运行 git --version 验证

（2）执行 Claude Code 安装命令

Git 确认安装完毕后，在管理员 PowerShell 中输入以下命令并回车：

# 安装稳定版本（推荐生产环境使用）

# 安装指定版本号
# & ([scriptblock]::Create((irm https://claude.ai/install.ps1))) 1.0.58

安装命令执行完成后，终端将显示如下输出：

  • √ Claude Code successfully installed! — 安装成功确认

警告原文的含义：Claude 已经安装成功了，但临门一脚还没踢进去 —— 它的安装路径还没有被添加到系统的"环境变量（PATH）"中。

（4）手动添加环境变量（Windows 用户必经之路）

安装脚本不会自动将 claude.exe 所在目录添加到系统 PATH，需要手动完成。操作路径为：按 Win + S 搜索 "环境变量" → 打开 "编辑账户的环境变量" → 在上方 "用户变量" 区域选中 Path → 点击 "编辑" → 点击 "新建" → 粘贴路径 C:\Users\snowb\.local\bin（将 snowb 替换为你的实际用户名）→ 逐层点击 "确定" 保存。

完成后务必关闭当前所有 PowerShell 或终端窗口，重新打开一个新的 PowerShell。 环境变量修改只对新打开的终端生效，不重启终端将无法识别 claude 命令。

（5）验证安装是否成功

重新打开 PowerShell 后，运行以下命令验证：

claude --version

如果能看到版本号（如 2.1.50），就说明一切搞定了！

如果仍提示"claude 不是内部或外部命令"，请检查：

  1. PATH 是否添加到了 "用户变量" 中的 Path（不是系统变量）
  2. 是否已完全关闭旧 PowerShell 并重新打开了新窗口
  3. 路径是否正确（注意将 snowb 换成你的实际用户名）

（6）配置 Git Bash 路径（仅在报错时需要）

如果运行 claude 时出现 "requires git-bash" 错误，说明 Claude Code 找不到 Git Bash，需要手动指定路径：

# 临时设置（仅当前会话有效）
$env:CLAUDE_CODE_GIT_BASH_PATH="C:\Program Files\Git\bin\bash.exe"

# 永久设置（通过系统属性 → 环境变量添加）
# 变量名: CLAUDE_CODE_GIT_BASH_PATH
# 变量值: C:\Program Files\Git\bin\bash.exe

方式二：WSL 安装

对于已经在使用 WSL（Windows Subsystem for Linux）的开发者，你可以直接在 WSL 环境中安装 Claude Code，使用体验与原生 Linux 完全一致。WSL 1 和 WSL 2 均受支持。

# 步骤 1: 进入 WSL
wsl

# 步骤 2: 执行 Linux 原生安装脚本
curl -fsSL https://claude.ai/install.sh | bash

使用 WSL 安装时需要特别注意以下几个常见陷阱：

陷阱一：Node/npm 路径冲突。如果你在 Windows 和 WSL 中都安装了 Node.js，WSL 可能会使用 Windows 的 npm（路径以 /mnt/c/ 开头），导致安装失败。运行 which npm 和 which node 确认它们指向 Linux 路径（以 /usr/ 开头）。如果不是，需要在 ~/.bashrc 中确保 nvm 正确加载。

陷阱二：跨文件系统性能问题。在 WSL 中访问 Windows 文件系统（/mnt/c/）的性能远低于访问 Linux 文件系统（/home/）。如果你的项目在 Windows 分区上，搜索结果可能不完整。建议将项目克隆到 Linux 文件系统中使用。

陷阱三：IDE 连接问题（WSL 2）。如果在 WSL 2 中使用 JetBrains IDE，可能会因为网络配置问题导致 IDE 无法检测到 Claude Code。解决方法是配置 Windows 防火墙规则或切换 WSL 2 的网络模式为 mirrored。

3.3 Linux 安装配置

完成了 Windows 平台的详细讲解之后，我们来看 Linux 平台。相比 Windows，Linux 的安装流程要简洁得多——毕竟 Claude Code 的底层就是为类 Unix 环境设计的，Linux 是它的「原生生态」。我们以最常用的 Ubuntu/Debian 系统为例，同时也会覆盖其他发行版的注意事项。

（1）确认系统环境

在安装之前，我们需要确认系统版本和基本工具是否满足要求。Claude Code 要求 Ubuntu 20.04+ 或 Debian 10+。

# 查看系统版本
cat /etc/os-release

# 确认 curl 已安装（安装脚本需要）
curl --version

# 如果 curl 未安装，通过包管理器安装
sudo apt update && sudo apt install -y curl    # Ubuntu/Debian
sudo yum install -y curl                        # CentOS/RHEL

（2）执行原生安装脚本

Linux 上的安装只需要一行命令。安装脚本会自动检测你的系统架构（x86_64 / ARM64），下载对应的二进制文件，并放置在 ~/.local/bin/ 目录下。

# 安装最新版本（默认）
curl -fsSL https://claude.ai/install.sh | bash

# 安装稳定版本（生产环境推荐）
curl -fsSL https://claude.ai/install.sh | bash -s stable

# 安装指定版本
curl -fsSL https://claude.ai/install.sh | bash -s 1.0.58

（3）验证安装并配置 PATH

安装完成后，重新打开终端（或执行 source ~/.bashrc），验证 claude 命令是否可用。

# 重新加载 shell 配置
source ~/.bashrc   # 或 source ~/.zshrc

# 验证 claude 命令
claude --version

# 如果提示 command not found，手动将 ~/.local/bin 添加到 PATH
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc
source ~/.bashrc

# 运行健康检查
claude doctor

如果一切顺利，claude --version 会输出当前安装的版本号，claude doctor 会显示安装类型和系统状态的全面检查报告。

（4）（可选）Homebrew 备选安装

如果你的 Linux 系统上已经安装了 Homebrew（Linuxbrew），也可以通过 Homebrew 安装 Claude Code。

# 通过 Homebrew 安装
brew install --cask claude-code

# 后续更新（Homebrew 不会自动更新，需手动执行）
brew upgrade claude-code

Alpine Linux 特别说明：如果你使用的是 Alpine Linux 或其他基于 musl/uClibc 的发行版，原生安装程序需要额外的依赖项。你需要先安装 libgcc、libstdc++ 和 ripgrep：

apk add libgcc libstdc++ ripgrep

并设置环境变量 USE_BUILTIN_RIPGREP=0 以使用系统安装的 ripgrep 替代内置版本。

3.4 macOS 安装配置

macOS 是 Claude Code 支持最完善的平台之一——Anthropic 的开发团队本身就大量使用 macOS，因此安装体验在三个平台中是最顺滑的。macOS 用户不需要像 Windows 那样额外安装 Git Bash，系统自带的 Terminal 和 Bash/Zsh 就是最佳运行环境。

（1）确认系统版本

Claude Code 要求 macOS 13.0（Ventura）或更高版本。同时支持 Intel 和 Apple Silicon（M1/M2/M3/M4）架构。

# 查看 macOS 版本

# 查看处理器架构
uname -m
# 输出 arm64 表示 Apple Silicon, x86_64 表示 Intel

（2）执行原生安装（推荐）

打开 Terminal（终端），执行以下命令。安装脚本会自动检测你的 Mac 是 Intel 还是 Apple Silicon 架构，下载对应的二进制文件。安装完成的二进制文件经过 Apple 公证（Notarized），并由「Anthropic PBC」签名，确保安全性。

# 安装最新版本（默认）

# 安装稳定版本

（3）验证安装

安装完成后，与 Linux 一样，验证命令是否可用：

# 验证版本
claude --version

# 健康检查
claude doctor

macOS 用户通常不会遇到 PATH 问题，因为 ~/.local/bin 在大多数 macOS 配置中已经包含在默认 PATH 中。如果确实遇到了，添加方式与 Linux 相同。

（4）（可选）通过 Homebrew 安装

macOS 上的 Homebrew 是最流行的包管理器。如果你更习惯通过 Homebrew 管理软件，也可以使用它来安装 Claude Code。 

# 安装

# 更新

（5）（可选）安装桌面应用

除了 CLI 之外，macOS 用户还可以安装 Claude Code 的桌面应用程序。桌面应用提供了可视化的 Diff 查看、多会话并行管理和云会话启动等功能，适合喜欢图形界面的用户。

访问官方下载链接下载 macOS 版本（通用版，同时支持 Intel 和 Apple Silicon）的 .dmg 文件，双击打开后将应用拖入 Applications 文件夹即可。启动应用后登录你的 Claude 账户，点击「代码」标签即可开始使用。

6. 在 Cursor 中使用 Claude Code 插件

前面几章我们完成了 Claude Code 在终端中的完整安装与配置流程。事实上，Claude Code 还提供了一个专为编辑器设计的使用路径——以插件形式嵌入 VS Code 或 Cursor 中直接运行。对于日常在 Cursor 里开发的同学来说，这意味着无需在终端和编辑器之间来回切换：你可以直接在编辑器侧边打开 Claude Code 面板，享受它的 Agent 编程能力，同时保留 Cursor 原有的补全与对话体验。

本章将完整演示如何在 Cursor 中安装并启用 Claude Code for VS Code 插件，以及如何通过 CC-Switch 的一键同步功能，让插件自动跟随当前激活的供应商配置（如 MiniMax-M2.5 或官方 Claude）——全程不需要手动编辑任何配置文件。整个流程分为三步：安装插件、在 CC-Switch 中开启同步、在 Cursor 中打开并验证。

步骤一：在 Cursor 中安装 Claude Code for VS Code 插件

由于 Cursor 基于 VS Code 内核构建，它完全兼容 VS Code 扩展市场中的所有插件。我们只需在扩展面板中搜索并安装官方的 Claude Code for VS Code 插件即可。本步骤的判据是：扩展面板中出现 Claude Code for VS Code 条目，且状态为已安装（显示 Disable / Uninstall 按钮）。

（1）在扩展面板中搜索并安装插件

打开 Cursor，点击左侧活动栏的扩展图标（或按 Ctrl+Shift+X），在搜索框中输入 Claude。在搜索结果中找到 「Claude Code for VS Code」（发布者：Anthropic，安装量 310 万+），点击 Install 完成安装。

安装完成后，插件卡片右侧会出现 Disable 和 Uninstall 按钮（替换原来的 Install），表示插件已成功安装并激活。右键点击插件条目，可以看到包含 Settings（设置）在内的上下文菜单——这是后续配置的入口，但使用 CC-Switch 时不需要在这里手动修改任何参数。

步骤二：在 CC-Switch 中开启「应用到 Claude Code 插件」

插件安装完成后，回到 CC-Switch。CC-Switch 内置了一项专为 VS Code / Cursor 插件设计的同步功能——只需在设置中开启一个开关，此后每次在 CC-Switch 中切换供应商时，插件所使用的模型和 API 配置都会自动同步更新，无需手动修改任何 JSON 配置文件或环境变量。本步骤的判据是：CC-Switch 设置页中「应用到 Claude Code 插件」开关处于开启状态（绿色）。

（1）点击 CC-Switch 主界面左上角的齿轮图标进入设置

打开 CC-Switch，在主界面左上角找到齿轮图标（设置按钮），点击进入设置页面。

（2）开启「应用到 Claude Code 插件」开关

进入设置页面后，在「通用」标签下找到 「应用到 Claude Code 插件」 选项，将右侧开关拨到开启状态（显示绿色）。该选项的说明文字为：「开启后 Vscode Claude Code 插件的供应商将随本软件切换」——这正是我们需要的效果。

开关开启后，CC-Switch 会立即将当前激活供应商的配置写入 Claude Code for VS Code 插件所读取的配置路径。后续每次在 CC-Switch 主界面点击某个供应商的「启用」按钮，插件的模型配置也会同步刷新，真正做到「一处切换，多处生效」。

步骤三：在 Cursor 中打开 Claude Code 面板并验证效果

完成上述两步后，回到 Cursor，打开 Claude Code 面板，验证插件是否已正确接入当前激活的供应商。本步骤的最终判据是：在 Cursor 的 Claude Code 面板中提问，能够收到正常回复。

（1）通过编辑器顶部菜单打开 Claude Code 面板

在 Cursor 中，点击右上角的 ...（更多操作）按钮，在弹出的下拉菜单中找到并点击 「Claude Code: Open」，即可在编辑器中打开 Claude Code 交互面板。

如果在菜单中找不到「Claude Code: Open」选项，请确认插件已正确安装且处于启用状态。也可以通过命令面板（Ctrl+Shift+P）搜索 Claude Code: Open 来打开。

（2）在 Claude Code 面板中发起对话，验证接入效果

Claude Code 面板在编辑器中打开后，在底部输入框中输入「你好，请问你是什么模型？」并发送。如果能够收到正常回复，说明插件已成功接入 CC-Switch 当前激活的供应商配置。

至此，在 Cursor 中使用 Claude Code 插件的完整配置流程已全部完成。后续如需切换底层模型（例如从官方 Claude 切换到 MiniMax），只需在 CC-Switch 主界面点击对应供应商的「启用」按钮，插件会自动同步，无需重启 Cursor 或做任何额外操作。

使用体验说明：在 Cursor 中，Claude Code 以独立对话面板的形式运行，与 Cursor 自身的 AI 补全和 Composer 功能相互独立、互不干扰。你可以同时使用 Cursor 的原生补全能力进行日常快速编辑，在需要执行复杂 Agent 任务（跨文件重构、Git 操作、测试修复等）时切换到 Claude Code 面板，充分发挥两者各自的优势。
