## 五、自定义 Skills 加载机制

Skills 有 **三种注入方式**，按场景选择：

| 方式 | 注册位置 | 用途 | 触发时机 |
|------|----------|------|----------|
| **Manifest `skills` 字段** | `openclaw.plugin.json` | Skill markdown 文件注入 agent system prompt | Agent 启动时自动加载 |
| **`agentTools`** | `ChannelPlugin.agentTools` | Channel 专属 AI 可调用工具 | Agent 运行时按需调用 |
| **`api.registerTool()`** | `register(api)` 中 | 全局 AI 工具（跨 channel） | Agent 运行时按需调用 |
| **`api.registerCommand()`** | `register(api)` 中 | 用户 `/command` 直接执行，绕过 LLM | 用户输入命令时 |
| **`agentPrompt.messageToolHints`** | `ChannelPlugin.agentPrompt` | 给 agent 的上下文提示 | 每次 prompt 构建时 |

## 六、关键约束与易踩坑

1. **`AccountId` 一致性**：inbound `MsgContext.AccountId`、session key、outbound target 三处的 accountId 必须完全一致
2. **Inbound 必须走 `finalizeInboundContext`**：不要跳过归一化，否则丢失 `CommandAuthorized` default-deny 等安全语义
3. **Outbound 必须走 `deliverOutboundPayloads`**：不要私自发消息，否则丢失 queue/hook/mirror 能力
4. **Command 命名限制**：`^[a-z][a-z0-9_-]*$`，且不能与 [保留命令](src/plugins/commands.ts) 冲突（help/status/stop/reset/config 等约 30 个）
5. **Tool 命名**：不能与 core tools 冲突，可通过 `tools.allow` 配置白名单
6. **`package.json` 依赖**：runtime deps 放 `dependencies`；`openclaw` 放 `devDependencies` 或 `peerDependencies`，不要用 `workspace:*`
7. **配置白名单**：非 bundled 插件需要在 `plugins.allow` 中显式声明才能加载

---

## 七、端到端加载链路

```
discoverOpenClawPlugins()
  → 扫描 extensions/my-channel/openclaw.plugin.json
  → loadPluginManifestRegistry() 校验 manifest
  → loadOpenClawPlugins()
    → 评估 allow/deny + config schema
    → jiti 加载 index.ts
    → 调用 register(api)
      → api.registerChannel() → registry.channels[]
      → api.registerCommand() → registry.commands[]
      → api.registerTool() → registry.tools[]
    → resolvePluginSkillDirs() → 发现 skills/ 目录
  → setActivePluginRegistry() 激活
  → listChannelPlugins() 排序
  → startChannels() → plugin.gateway.startAccount()
  → channel-health-monitor 持续巡检
```

总结：新建 Channel + 自定义 Skills 的核心是 **一个 manifest + 一个 register 函数 + ChannelPlugin 契约实现**，Skills 通过 manifest 的 `skills` 字段声明目录、通过 `agentTools` 提供工具、通过 `registerCommand` 提供用户命令，三者协同构成完整的 channel 能力面。
